Language & Framework Independent

Challenges of Microservices التحديات والعيوب
⚠️ Challenges — التحديات
  • Complexity (التعقيد): A microservices application is more complex as compared to the equivalent monolithic application. السيستم بيبقى معقد جداً مقارنة بـ Monolithic.
  • Development & Testing (التطوير والاختبار): Building and testing a service that relies on other services need domain understanding and refactoring them can be difficult. صعب تبني وتختبر Service تعتمد على Services تانية — محتاج فهم كبير للنظام.
  • Lack of Governance (غياب الحوكمة): The decentralized approach leads to so many problems like maintenance because of many different languages and frameworks. كل Team بيستخدم لغة مختلفة وFramework مختلف — ده يعمل مشاكل إدارة وصيانة.
  • Network Congestion & Latency (بطء الشبكة): The use of many small services can result in additional latency because of inter-service communication and a long chain of service dependencies. الحل: design APIs carefully and use asynchronous communication patterns.
  • Data Integrity (تكامل البيانات): Each microservice is responsible for its own data persistence. As a result, data consistency can be a challenge. كل Service ليها DB خاص بيها — صعب توحيد الداتا بين الخدمات.
  • Management (الإدارة): To be successful with microservices requires a mature DevOps culture. Correlated logging across services can be challenging for a single user operation. لو حصل Error — مين السبب؟ Service A ولا B ولا C؟ صعب تتبع المشكلة.
  • Versioning (إدارة الإصدارات): Be careful while updating a service. It must not break services that depend on it. Without careful design, you might have problems with backward or forward compatibility. التحديثات خطر — لازم الخدمات تبقى متوافقة مع بعض.
  • Skillset (المهارات): Microservices are highly distributed systems. So need a skilled and experienced team to implement it. محتاج Team خبرة عالية وفاهم Distributed Systems.

When to Use Microservices إمتى أستخدم Microservices؟
01
High Release Velocity
Large applications that require a high release velocity — سيستم كبير وعايز تنزل Features بسرعة. تعدل Service واحدة بس بدل ما تعدل السيستم كله.
02
Highly Scalable
Complex applications that need to be highly scalable — محتاج تكبر جزء معين بس. مثال: Service الدفع عليها ضغط عالي → تكبرها وحدها.
03
Rich Domains / Subdomains
Applications with rich domains or many subdomains — السيستم متقسم طبيعي لأجزاء كتير: Users / Orders / Payments / Shipping.
04
Small Development Teams
An organization that consists of small development teams — كل Team يمسك Service مستقلة. Team 1 → Users, Team 2 → Orders, Team 3 → Payments.

Design Patterns for Microservices Design Patterns = حلول جاهزة لمشاكل متكررة في تصميم السيستم
🔀 Decomposition Patterns تقسيم السيستم
Decompose by Business Capability
نقسم السيستم حسب الوظائف: Payments / Orders / Users — كل واحدة تبقى Service لوحدها
Decompose by Subdomain
من Domain Driven Design (DDD) — نقسم حسب Subdomain. مثال: Ecommerce → Orders / Shipping / Inventory
Strangler Pattern
نحول السيستم القديم (Monolith) لـ Microservices تدريجي — بدل ما نكسره مرة واحدة. زي الشجرة اللي بتخنق القديمة (Strangler Fig)
📡 Communication Patterns التواصل بين الخدمات
Request / Response Pattern
زي HTTP: Client يطلب وService ترد — Sync (مباشر)
Messaging Pattern
باستخدام Queue: Service تبعت Message، Service تانية تستقبل — Async (مش فوري)
Event Driven Pattern
كل حاجة Events: Order Created / Payment Done — Services تreact على الأحداث
🔌 Integration Patterns ربط الخدمات
API Gateway Pattern
بوابة واحدة لكل الطلبات — بدل ما الClient يكلم كل Service يكلم Gateway بس
Aggregator Pattern
Service تجمع data من Services تانية. مثال: Profile page → يجيب من User + Orders + Payments
Client-Side UI Composition
الFrontend نفسه يكلم كل Service ويجمع البيانات — عكس Aggregator
🗄️ Database Patterns إدارة البيانات
Database per Service ✅
كل Service ليها Database — الأفضل
Shared Database ❌
كل Services على نفس DB — غلط غالباً (tight coupling)
CQRS Pattern
نفصل: Read (Query) عن Write (Command) — لتحسين الأداء
Saga Pattern
لإدارة Transactions بين Services — بدل Transaction واحدة: سلسلة خطوات
🚀 Deployment Patterns تشغيل الخدمات
Multiple Service Instances per Host
أكتر من نسخة من Service على نفس السيرفر — للـ Scaling
Service Instance per Host
كل Service لوحدها على سيرفر — عزل كامل
Serverless Deployment
بدون سيرفرات — زي AWS Lambda. بتدفع حسب الاستخدام
Service Deployment Platform
Platforms زي Kubernetes — لإدارة الخدمات
👁️ Observability Patterns مراقبة السيستم
Log Aggregation
كل Logs تتجمع في مكان واحد
Performance Metrics
متابعة: CPU / Memory / Response time
Distributed Tracing
تتبع الطلب وهو ماشي بين Services
Health Check
هل Service شغالة ولا لا؟
⚙️ Cross-Cutting Concern Patterns حاجات مشتركة
Externalized Configuration
نخلي الإعدادات برا الكود — زي config files أو environment variables
Service Discovery Pattern
Services تلاقي بعض تلقائي — زي Eureka
Circuit Breaker Pattern
لو Service وقعت — نوقف الCalls ليها عشان نحمي السيستم

Service Oriented Architecture — SOA يعني إيه SOA أصلاً؟
التعريف الرسمي: Service-oriented architecture (SOA) is an enterprise-wide approach to software development of application components that takes advantage of reusable software components, or service.

بالمصري: SOA = طريقة تصميم سيستم على مستوى الشركة كلها باستخدام Services قابلة لإعادة الاستخدام. كل Service تقدر تستخدمها في أكتر من سيستم — زي LEGO 🧱
🏢
Enterprise-Wide
على مستوى الشركة كلها

مش مشروع صغير — ده نظام كبير بيستخدم في شركات كبيرة. كل السيستم بتاع الشركة مبني على Services مشتركة.

♻️
Reusable Services
خدمات قابلة لإعادة الاستخدام

It uses software components called services to create business applications. تعمل Service مرة وتستخدمها في كذا سيستم.

🌍
Cross-Platform & Cross-Language
بيشتغل على أي منصة ولغة

Services can also communicate with each other across platforms and languages. واحدة Java وواحدة Python — عادي يتكلموا.

🔗
Combine for Complex Tasks
تجمعهم لأداء مهام معقدة

Developers use SOA to reuse services in different systems or combine several independent services to perform complex tasks. مثال: عملية شراء = User + Order + Payment + Shipping.

SOA Roles الدورين الأساسيين في SOA — There are two major roles within SOA
🏭
Service Provider
مزود الخدمة — اللي بيقدم وينشر الخدمة
1
Maintainer: The service provider is the maintainer of the service — اللي بيصونها ويحدثها
2
Publish: يحط الخدمة في Registry — the provider can publish them in a registry
3
Service Contract: يعمل عقد بيوضح: nature of the service / how to use it / the requirements / the fees charged
مثال: Stripe بتقدم Payment API — بتشرح استخدامها وشروطها وسعرها
💻
Service Consumer
مستخدم الخدمة — اللي بيدور ويستخدم الخدمة
1
Locate: يدور على الخدمة في الRegistry — can locate the service metadata in the registry
2
Develop: يكتب كود — develop the required client components
3
Bind & Use: يربط ويستخدمها — to bind and use the service
مثال: أنت كمبرمج: دخلت شفت الـ API، قريت الـ docs، كتبت كود، استخدمتها

Enterprise Service Bus — ESB القلب بتاع السيستم ❤️ — الوسيط الذكي بين كل الخدمات
التعريف الرسمي: An architectural pattern that centralized software component performs integrations between applications. It performs transformations of data models, handles connectivity and messaging, performs routing, converts communication protocols and potentially manages the composition of multiple requests.

بالمصري: ESB = برنامج واحد في النص بيشبّك كل السيستم ببعض. زي موظف سنترال: حد يطلب خدمة → يوجهه للصح، يحول الرسالة، يظبط التواصل.

المعنى العميق (من Schmidt et al., IBM Systems Journal 2005): The ESB manages and exploits meta-data describing interaction endpoints as well as the domain models used to describe the capabilities of those endpoints; it supports configuration of links that bridge between capabilities demanded by service requestors and those offered by service providers, dynamically matching requestors with providers and in the process establishing and enacting contracts between those interaction endpoints.
Meta-data: بيانات عن البيانات — اسم الخدمة، عنوانها، بتشتغل إزاي. ESB عارف كل Service فين (endpoint) وبتعمل إيه. بيربط اللي طالب الخدمة باللي بيقدمها بشكل أوتوماتيك (dynamically matching). بيتأكد إن كل Service ملتزمة بالقواعد (establishing and enacting contracts).
1️⃣ Route Messages Between Services
يوجه الرسائل — "الطلب ده يروح لمين؟"
2️⃣ Monitor & Control Routing
Monitor and control routing of message exchange between services — يراقب ويتحكم في مسار الرسائل
3️⃣ Resolve Contention
Resolve contention between communicating service components — لو Services كتير بتطلب نفس الحاجة ينظمهم. زي المرور 🚦
4️⃣ Deployment & Versioning
Control deployment and versioning of services — يدير تشغيل الخدمات ويتحكم في نسخها (v1 / v2)
5️⃣ Event Handling
Provide commodity services like event handling — التعامل مع الأحداث
6️⃣ Data Transformation & Mapping
Data transformation and mapping — يحول شكل الداتا. مثال: JSON ↔ XML
7️⃣ Message & Event Queuing / Sequencing
Message and event queuing and sequencing — يرتب الرسائل في طابور وينظم التنفيذ
8️⃣ Security & Exception Handling
Security or exception handling — الأمان والتعامل مع الأخطاء
9️⃣ Protocol Conversion
Protocol conversion — يحول البروتوكول. مثال: HTTP ↔ SOAP
🔟 Quality of Service
Enforcing proper quality of communication service — سرعة، استقرار، دقة

SOA Service Types & Components أنواع الخدمات ومكونات كل Service

SOA PROVIDES FOUR DIFFERENT SERVICE TYPES — SOA بيقدم 4 أنواع خدمات

⚙️
Functional Services
خدمات وظيفية / Business Services

Critical for business applications — الأساسية للبزنس. هي الخدمات اللي بتنفذ العمليات البزنس الفعلية.

🏗️
Enterprise Services
خدمات المؤسسة

Serve to implement functionality — بتخدم تنفيذ الوظائف على مستوى المؤسسة.

📱
Application Services
خدمات التطبيق

Used to develop and deploy apps — بتتستخدم لتطوير ونشر التطبيقات.

🔐
Infrastructure Services
خدمات البنية التحتية

Instrumental for backend processes like security and authentication — للعمليات الخلفية زي الأمان والمصادقة.

EACH SERVICE CONSISTS OF THREE COMPONENTS — مكونات أي Service

🖥️
COMPONENT 1
Interface
الواجهة — defines how a service provider will execute requests from a service consumer. بتقولك: هتكلم الخدمة إزاي؟ إيه الـ API، إيه الـ inputs & outputs.
📜
COMPONENT 2
Contract
العقد — defines how the service provider and service consumer should interact. القوانين بين الطرفين: الشروط، الفورمات، أحيانًا السعر.
💻
COMPONENT 3
Implementation
التنفيذ — which is the service code. الكود الحقيقي اللي شغال: Java / Python / PHP — اللوجيك كله هنا.

SOA ADVANTAGES & DISADVANTAGES

✅ Advantages of SOA
  • Service Reusability: In SOA, applications are made from existing services. Thus, services can be reused to make many applications. تعمل Service مرة وتستخدمها في كذا سيستم.
  • Easy Maintenance: As services are independent of each other they can be updated and modified easily without affecting other services. كل Service لوحدها → تعدل فيها بسهولة.
  • Platform Independent: SOA allows making a complex application by combining services picked from different sources, independent of the platform. Java + Python عادي يشتغلوا مع بعض.
  • Availability: SOA facilities are easily available to anyone on request. متاحة بسهولة لأي حد عنده access.
  • Reliability: SOA applications are more reliable because it is easy to debug small services rather than huge codes. كل Service صغيرة → سهل تكتشف الغلط.
  • Scalability: Services can run on different servers within an environment, this increases scalability. تشغل Services على سيرفرات مختلفة → السيستم يكبر بسهولة.
  • Faster Time to Market: بتستخدم Services جاهزة → تطلع المنتج بسرعة 🚀
❌ Disadvantages of SOA
  • ESB Bottleneck: ESB can be bottlenecks — ESB ممكن يبقى بطيء ويعطل السيستم كله. عنق زجاجة.
  • Single Point of Failure: ESB can be a single point of failure — لو ESB وقع 💥 السيستم كله يقع.

SOA vs Microservices الفرق والتشابه
التشابه (Similarities): Some critics of the microservices architecture claim it's nothing new — it's SOA. At a very high level, there are some similarities. Both are architectural styles that structure a system as a set of services. الاتنين بيبنوا السيستم باستخدام Services وكل Service مستقلة وبيكلموا بعض.

الفرق الأساسي: The main distinction between the two approaches comes down to scope. To put it simply, service-oriented architecture (SOA) has an enterprise scope, while the microservices architecture has an application scope. (IBM Cloud)
الجانب 🟠 SOA 🟢 Microservices
Scope Enterprise-wide scope (الشركة كلها) Application scope (التطبيق)
Communication Smart pipes — ESB (ذكي ومركزي). Managing and coordinating through ESB. ESB can become a single point of failure (SOF). Dumb pipes — Direct communication. كل Service مستقلة بتتواصل مباشرة.
Protocols Heavyweight / Heterogeneous: SOAP, AMQP (Advanced Messaging Queuing Protocol), MSMQ (Microsoft Messaging Queuing) Lightweight: HTTP/REST (Representational State Transfers), JMS (Java Messaging Service), gRPC
Data Storage Global data model & Shared Database — database storage is shared between all services Database per service — كل Service ليها DB خاص
Service Size Larger — can range from small, specialized services to enterprise-wide services Smaller — highly specialized, each designed to do one thing very well
Service Granularity متنوع: من صغير لـ enterprise-wide Highly specialized — كل service بتعمل حاجة واحدة بس
Speed Slower — by leveraging shared architecture, SOAs simplify development and troubleshooting. But this tends to make SOAs operate more slowly. Faster — minimize sharing in favor of duplication. أسرع بكتير
Architecture مركزي — Centralized (ESB) موزع — Distributed & Decentralized
Interoperability More open to heterogeneous messaging protocols — SOAP, AMQP, MSMQ Lightweight messaging — HTTP/REST, JMS

Summary — الخلاصة الكاملة للامتحان أهم النقاط اللي لازم تحفظها

Microservices

سيستم متقسم لخدمات صغيرة مستقلة. كل Service ليها Database + Team + CI Pipeline. التواصل بينهم REST APIs. مميزات: Flexibility, Scalability, Fault isolation.

Challenges

Complexity, Development & Testing, Lack of Governance, Network Latency, Data Integrity, Management, Versioning, Skillset — محتاج Team خبرة عالية.

When to Use

Large apps with high release velocity + Complex & Scalable apps + Rich domains/subdomains + Organizations with small independent teams.

Design Patterns

Decomposition, Communication, Integration, Deployment, Database, Observability, Cross-Cutting Concerns — كل واحدة بتحل مشكلة معينة.

SOA

Enterprise-wide approach using reusable services. Provider publishes + Consumer uses. Shared DB. ESB as central communication hub. 4 types: Functional, Enterprise, Application, Infrastructure.

ESB Duties

Routing + Monitoring + Resolve Contention + Deployment/Versioning + Event Handling + Data Transformation + Queuing + Security + Exception Handling + Protocol Conversion + Quality of Service.

Service = 3 Components

Interface (إزاي تستخدمها) + Contract (القوانين) + Implementation (الكود)

SOA vs Microservices

SOA: Enterprise scope + ESB (Smart Pipes) + SOAP + Shared DB + Larger + Slower. Microservices: App scope + Direct (Dumb Pipes) + REST + DB per Service + Smaller + Faster.
