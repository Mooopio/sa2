📌 Case Study (Applies to All Questions)

You are the lead architect of a large-scale e-commerce platform called ShopFlow. The system has been redesigned from a monolithic application into a microservices-based architecture. The platform includes four core services:

Orders Service Payments Service Inventory Service Shipping Service

Each service owns its own private database. The system must handle high throughput, ensure data consistency, and remain available 24/7. Business transactions frequently span multiple services.

01
5 marks 🔥 High Probability Microservices vs Monolithic
ShopFlow started as a monolithic application. Explain the problems (monolithic hell) that led the team to migrate to microservices. What specific advantages did the microservices architecture bring to ShopFlow?
▼
Problems with Monolithic (Monolithic Hell)
  • Slow Development: As the ShopFlow codebase grew, adding features became complex and time-consuming. Multiple Scrum teams were forced to work on the same codebase, causing conflicts and delays.
  • No Independent Scaling: ShopFlow could not scale only the Payments service during peak hours — the entire monolith had to be scaled, wasting resources.
  • Long Deployment Cycle: Every change required the entire application to be rebuilt, tested, and deployed as one unit — making releases slow and risky.
  • Technology Lock-in: The team was forced to use a single technology stack, even if a specific service (e.g., Inventory search) would benefit from a different database like ElasticSearch.
  • Poor Fault Isolation: A bug in the Shipping module could crash the entire ShopFlow application, affecting Orders and Payments too.
Advantages Gained from Microservices
  • Independent Deployment: Payments, Orders, and Inventory can each be deployed separately without affecting each other — zero downtime upgrades.
  • Selective Scaling: During Black Friday, only the Orders and Payments services need to scale horizontally.
  • Technology Diversity: Inventory can use ElasticSearch for text search, while Orders uses PostgreSQL — each service picks the best tool.
  • Better Fault Isolation: If Shipping service fails, Orders and Payments continue running normally.
  • Continuous Delivery: Small teams can release their services independently, supporting CI/CD pipelines.
02
5 marks 🔥 High Probability CQRS Pattern
How would implementing CQRS improve performance and scalability in the Orders service of ShopFlow? What trade-offs might this introduce?
▼
How CQRS Improves Performance
CQRS (Command Query Responsibility Segregation) separates the Orders service database into two models: a Write Model (Command) for creating/updating orders and a Read Model (Query) for fetching order data.
  • Independent Scaling: If ShopFlow has 100x more read requests (customers checking order status) than writes (placing orders), the read database can scale out independently using NoSQL (e.g., MongoDB) optimized for fast lookups.
  • Eliminates Complex Joins: The read model stores pre-computed, denormalized data (e.g., order + customer name + product details in one document), removing slow 10-table JOIN queries that could lock the database.
  • Optimized Databases: Use a relational DB (PostgreSQL) for commands (complex business rules, validations) and a NoSQL DB for queries (fast reads).
  • Better Security: Read and write endpoints can have different access control policies.
Client → POST /orders → [Command Handler] → Write DB (PostgreSQL) ↓ Domain Event [Event Broker (Kafka)] ↓ [Query Model Updater] → Read DB (MongoDB) Client → GET /orders/123 → [Query Handler] → Read DB (MongoDB)
Trade-offs Introduced

Cons / Challenges

  • Eventual consistency — read model may lag behind the write model by milliseconds/seconds
  • Increased complexity — two separate models, event propagation, and sync logic to maintain
  • Data duplication between read and write stores
  • Not suitable for simple CRUD applications where overhead exceeds benefit

Pros Recap

  • Read and write sides scale independently
  • Optimized performance per workload type
  • Enables full audit trails when combined with Event Sourcing
  • Cleaner separation of concerns for development teams

5 marks 🔥 High Probability Database per Service
Explain how using a separate database for each service in ShopFlow promotes microservice autonomy (independence). What are the challenges in querying data across services?
▼
How It Promotes Autonomy
  • Loose Coupling: Changes to the Inventory database schema (e.g., adding a new column) do not require changes in Orders or Payments — each service evolves independently.
  • Technology Freedom: Orders uses PostgreSQL, Inventory uses ElasticSearch for text search, Shipping uses a graph DB (Neo4j) for route optimization — each picks the best fit.
  • Independent Deployment: The Payments team can update and redeploy their service without coordinating with other teams.
  • Failure Isolation: If the Shipping database goes down, Orders and Payments databases continue to operate normally.
  • Independent Scaling: The Orders database can be scaled separately during checkout spikes without touching other databases.
Challenges in Cross-Service Data Queries
  • No Native JOINs: You cannot JOIN an Orders table with a Customers table because they live in separate databases — potentially on separate servers.
  • Distributed Transactions: Maintaining data consistency across Orders + Payments + Inventory requires complex patterns (Saga) because 2PC is impractical.
  • Data Duplication: To avoid cross-service calls, services may store copies of other services' data (e.g., Orders stores customer name), leading to sync challenges.
  • Increased Query Complexity: To display an order summary (order + product + customer + shipping status), the API Gateway must query 4 services and aggregate the results (Aggregator Pattern).
  • CAP Theorem Constraints: With distributed databases, the system must choose between Consistency and Availability when a Partition occurs.

5 marks 🔥 High Probability Saga Pattern
Design data synchronization between the Inventory and Orders services in ShopFlow. Compare using Choreography vs. Orchestration Saga patterns for this scenario.
▼
Why Saga is Needed
When a customer places an order on ShopFlow, the transaction spans: (1) Orders service creates the order, (2) Payments service charges the customer, (3) Inventory service reserves stock. These are 3 separate databases — 2PC is not viable. The Saga pattern uses a sequence of local transactions with compensating transactions on failure.
Choreography Approach (Event-Driven)
Orders → OrderCreatedEvent → [Kafka] ↓ Payments listens → charges customer → PaymentCompletedEvent → [Kafka] ↓ Inventory listens → reserves stock → InventoryReservedEvent ✓ FAILURE: Payment fails → PaymentFailedEvent → Orders listens → cancels order (compensation)
Orchestration Approach (Centralized)
CreateOrderSaga (Orchestrator) → Orders: createOrder() → PENDING ✓ → Payments: reserveCredit() → PAID ✓ → Inventory: reserveStock() → RESERVED ✓ FAILURE: creditLimitExceeded → Orchestrator → Orders: rejectOrder() [compensation]
Comparison

Choreography

  • No single point of failure
  • Fully decoupled services
  • Natural for event-driven systems
  • Hard to trace the full flow
  • Complex failure handling per service

Orchestration

  • Clear, centralized business logic
  • Easy to debug and monitor
  • Orchestrator = single point of failure
  • Services are tightly coupled to orchestrator
  • Risk of "god service" anti-pattern
Recommendation for ShopFlow: Use Orchestration for the critical order-payment-inventory flow (clear, auditable) and Choreography for less critical notifications (e.g., shipping updates, email alerts).
05
5 marks Shared Database
If all ShopFlow services shared a single database, what benefits and risks would this introduce, especially regarding team independence and system failures?
▼
Benefits of Shared Database
  • Simple ACID Transactions: A developer can use a single SQL transaction to update Orders and deduct Inventory atomically — no Saga needed.
  • Easy Cross-Service Queries: A single JOIN query can retrieve order + customer + inventory + shipping data without calling multiple services.
  • Simpler Operations: Only one database to back up, monitor, and maintain.
  • No Data Duplication: Single source of truth for all data.
Risks Introduced
  • Development Coupling: If the Orders team changes the schema of the Orders table, they must coordinate with the Inventory and Payments teams — slowing everyone down.
  • Runtime Coupling: A long-running query from the Payments service (e.g., monthly report) can lock tables and block the Orders service — causing downtime across the entire ShopFlow platform.
  • Single Point of Failure: If the shared database goes down, ALL services (Orders, Payments, Inventory, Shipping) fail simultaneously.
  • No Technology Flexibility: All services are forced to use the same database engine — Inventory cannot switch to ElasticSearch for full-text search.
  • Scalability Bottleneck: A single database cannot efficiently handle the very different scaling needs of all four services simultaneously.
Conclusion: Shared database is only recommended as a transitional pattern (Strangler Pattern migration) or for very small teams. For ShopFlow's scale, Database per Service is the correct approach.

5 marks 🔥 High Probability SOA vs Microservices
ShopFlow's CTO is comparing SOA and Microservices before the redesign. Compare the two architectures across: communication, interoperability, granularity, and speed. Which would you recommend for ShopFlow and why?
▼
SOA vs Microservices Comparison
Scope: SOA has an enterprise scope (integrates multiple enterprise applications via ESB), while Microservices has an application scope (decomposes a single application into small services).
  • Communication: SOA uses a central ESB (Enterprise Service Bus) as the communication backbone — the ESB becomes a single point of failure. Microservices communicate directly or through lightweight brokers (Kafka, RabbitMQ) with no central bottleneck.
  • Interoperability: SOA uses heavyweight protocols like SOAP, AMQP, MSMQ. Microservices prefer lightweight HTTP/REST and gRPC — simpler and faster.
  • Granularity: SOA services can be large (enterprise-wide services). Microservices are small and highly specialized — each does one thing well (e.g., only handles order creation).
  • Speed: SOA is slower due to the ESB layer and shared architecture overhead. Microservices minimize sharing, favoring duplication for speed.
  • Database: SOA shares a single database across all services. Microservices give each service its own database.
Recommendation
Microservices is recommended for ShopFlow because: (1) ShopFlow is a single application that needs high scalability, not enterprise-wide integration. (2) The team is small and organized around the four domains. (3) They need independent deployment, fault isolation, and technology diversity — all better served by Microservices than SOA.

5 marks 2PC vs Saga 🔥 High Probability
Why is the Two-Phase Commit (2PC) protocol not suitable for ShopFlow's microservices architecture? How does the Saga pattern address these problems?
▼
Why 2PC Fails for Microservices
  • Blocking Protocol: During Phase 1 (Prepare), all services (Orders, Payments, Inventory) lock their resources and wait. If the Inventory service is slow, the Orders service is blocked — reducing throughput dramatically.
  • Single Point of Failure: If the 2PC Coordinator (the central manager) crashes after sending COMMIT to Orders but before sending to Payments, Payments is stuck indefinitely — unsure whether to commit or rollback.
  • Tight Coupling: All services must be available simultaneously and synchronously respond — violating the independence principle of microservices.
  • No Broker Support: Kafka and RabbitMQ (the standard messaging tools in microservices) do not natively support distributed 2PC transactions.
  • Performance Bottleneck: Two synchronous round trips across the network for every transaction add significant latency in high-throughput e-commerce scenarios.
How Saga Solves These Problems
  • No Locking: Each service executes its own local ACID transaction and releases locks immediately — no waiting for other services.
  • Compensating Transactions: Instead of a global rollback, if Payments fails, a PaymentFailed event triggers a compensating transaction in Orders (cancel the order) — achieving eventual consistency.
  • Loose Coupling: Services communicate via events (Choreography) or commands (Orchestration) — they do not need to be available simultaneously.
  • Broker Compatible: Sagas are designed to work with Kafka and RabbitMQ natively.
Key difference: 2PC guarantees strong consistency but blocks the system. Saga guarantees eventual consistency but keeps the system available and loosely coupled — which is the right trade-off for ShopFlow.

5 marks Communication Patterns
ShopFlow services communicate internally. Compare synchronous (REST/Request-Response) and asynchronous (Messaging) communication patterns. When should each be used in ShopFlow?
▼
Synchronous Communication (REST / Request-Response)
  • How it works: The Orders service makes an HTTP call to the Payments service and waits for a response before proceeding.
  • Tight Runtime Coupling: Both services must be available at the same time — if Payments is down, Orders cannot complete.
  • Simple and Familiar: Easy to implement, test with tools like Postman or curl, no broker needed.
  • Best for: Real-time queries where the result is needed immediately — e.g., GET /orders/123 to display order status, or authenticating a user before allowing checkout.
Asynchronous Communication (Messaging / Event-Driven)
  • How it works: Orders service publishes an OrderCreatedEvent to Kafka. Payments service subscribes to the event and processes it independently — Orders does not wait.
  • Loose Coupling: Services don't need to be available simultaneously — Kafka buffers messages until the consumer is ready.
  • Higher Availability: ShopFlow continues accepting orders even if the Payments service is temporarily down.
  • Best for: Long-running workflows (order processing, payment, inventory reservation), notifications, and scenarios where eventual consistency is acceptable.
ShopFlow Design: Use REST for the customer-facing API layer (get order status, browse products). Use Messaging (Kafka) for internal inter-service workflows (order → payment → inventory → shipping pipeline).

5 marks DevOps & CI/CD
ShopFlow adopts DevOps practices. Explain the DevOps lifecycle stages and how CI/CD pipelines enable faster and more reliable deployment of microservices.
▼
DevOps Lifecycle Stages
  • Plan: Teams gather requirements and plan features using feedback from monitoring (e.g., slow checkout identified in monitoring → plan performance improvement sprint).
  • Code: Developers write small chunks of code in Git branches. ShopFlow teams use Git for version control.
  • Build: Jenkins pulls the latest code and builds it (Maven for Java services). Every commit triggers an automatic build.
  • Test: Automated testing via Selenium and unit tests run automatically — bugs are caught early before they reach production.
  • Release: The built artifact is approved for deployment — scheduling, change management.
  • Deploy: The Payments service is containerized with Docker and deployed automatically to staging then production without manual steps.
  • Operate: The service runs in production, serving real ShopFlow customers.
  • Monitor: Tools like Nagios, ELK Stack, or Splunk monitor service health, error rates, and performance — feeding back into Plan.
How CI/CD Enables Microservices Delivery
  • CI (Continuous Integration): Every code push is automatically built and tested — the Inventory team's changes are integrated into the main branch multiple times per day without breaking other services.
  • CD (Continuous Deployment): After tests pass, the new version of the Payments service is automatically deployed to production — no manual deployment meetings needed.
  • Independent Pipelines: Each of the 4 services (Orders, Payments, Inventory, Shipping) has its own CI/CD pipeline — enabling true independent deployment.
DevOps breaks the traditional "wall" between Developers and Operations teams — ShopFlow goes from monthly releases to multiple releases per day per service.

5 marks Scrum Framework
ShopFlow uses Scrum to manage development across its 4 service teams. Describe the key Scrum roles, ceremonies, and artifacts. How does Scrum support agile microservices development?
▼
Scrum Roles
  • Product Owner: Defines ShopFlow's feature priorities (e.g., "add one-click checkout" has highest business value), manages the Product Backlog, and accepts/rejects sprint results.
  • Scrum Master: Removes blockers for the Payments team (e.g., waiting for API documentation from the Orders team), shields the team from external interruptions, and ensures Scrum practices are followed.
  • Development Team: Cross-functional team of 5-9 members (developers, testers, DevOps engineers) who self-organize and build the increment each sprint.
Ceremonies & Artifacts
  • Sprint Planning: The Inventory team selects backlog items for the 2-week sprint, estimates tasks (1-16 hours each), and sets the Sprint Goal.
  • Daily Scrum (15 min stand-up): Each team member answers: What did I do yesterday? What will I do today? Is anything blocking me?
  • Sprint Review: Demo of completed features to stakeholders (e.g., demo of new stock reservation API).
  • Sprint Retrospective: Team reflects on Start/Stop/Continue — what worked, what didn't, what to improve next sprint.
  • Product Backlog: Prioritized list of all ShopFlow features — managed by the Product Owner.
  • Sprint Backlog: Items selected for the current sprint, with task-level estimates updated daily.
  • Burndown Chart: Visual showing remaining work vs. time — helps the team see if they're on track to finish the sprint.
Scrum enables microservices development because each service team operates as an autonomous Scrum team with its own backlog, sprint, and deployment pipeline — matching the independence principle of microservices architecture.

The Payments service in ShopFlow receives 10x more read requests (checking payment status) than write requests (processing payments). How would CQRS solve this imbalance? What consistency challenges arise?

How CQRS Solves the Imbalance

Compare CQRS with Kafka (manual event publishing) vs. CQRS with Kafka + CDC (Debezium). When would you choose each approach for keeping ShopFlow's Inventory read model in sync?
▼
CQRS + Kafka Only (Manual Event Publishing)
  • How: After the Inventory service updates stock, it explicitly publishes a StockUpdatedEvent to Kafka. A consumer updates the read DB.
  • Pro: Full control over event payload — can include rich business data (product name, category, supplier info).
  • Con: Risk of dual-write failure — if the DB write succeeds but Kafka publish fails, the read model is out of sync. Requires Outbox Pattern for reliability.

  • How: Inventory service only writes to PostgreSQL. Debezium watches the transaction log and automatically publishes change events to Kafka. A consumer updates the read DB.

  • Con: Events are low-level DB row changes — lose business context (why was stock reduced? — sale or damage?). Debezium + Kafka Connect adds infrastructure complexity.
  • Best for: Synchronizing read models from an existing DB without modifying the write service's code.

How would you design data synchronization between Inventory and Order services, given they each own their own database?

Design Approach: Event-Driven Synchronization
  • Step 1 — Order Placed: Orders service creates an order (state=PENDING) and publishes an OrderCreatedEvent to Kafka.

  • Idempotency: Inventory service must handle duplicate messages (if Kafka retries) without double-reserving stock — use idempotency keys.
Orders DB → OrderCreatedEvent → [Kafka Topic: orders] ↓ Inventory Service subscribes → checks stock in Inventory DB → reserves stock → InventoryReservedEvent → [Kafka] ↓ Orders Service subscribes → updates order state = CONFIRMED

A customer places an order on ShopFlow but the Payments service is temporarily down. Using a Choreographed Saga, explain step-by-step how the system handles this failure and maintains data consistency.
▼
Failure Handling with Choreographed Saga
  • Step 1: Orders service creates the order (state=PENDING) and publishes OrderCreatedEvent to Kafka.
  • Step 2: Payments service is DOWN — it cannot consume the event. Kafka buffers the message durably until Payments comes back online.
  • Step 3: When Payments recovers, it consumes the OrderCreatedEvent, attempts to charge the customer, and — if successful — publishes PaymentCompletedEvent.
  • Alternative — Payment Ultimately Fails: If Payments comes back but the credit card is declined, it publishes PaymentFailedEvent.
  • Compensation: Orders service listens to PaymentFailedEvent and executes compensating transaction: update order state to CANCELLED, release reserved inventory (publishes ReleaseInventoryEvent).
  • Key Insight: Unlike 2PC (which would have blocked the Orders service while waiting for Payments to respond), the Saga allowed Orders to continue accepting new orders — Kafka absorbed the unavailability.

5 marks Variant
The ShopFlow team is considering using a NoSQL database (MongoDB) for the Orders service instead of a relational DB. Explain the CAP theorem and how it influences this decision. What consistency trade-offs does MongoDB introduce compared to PostgreSQL?
▼
CAP Theorem
The CAP theorem states that a distributed system can guarantee at most TWO of the following three properties simultaneously:
  • Consistency (C): All nodes see the same data at the same time. Every read returns the most recent write.
  • Availability (A): Every request receives a response (success or failure) — the system is always up.
  • Partition Tolerance (P): The system continues operating even when network partitions (message loss between nodes) occur.
PostgreSQL vs MongoDB in Context
  • PostgreSQL (RDBMS): Achieves C + A (in non-distributed setup, no P). Guarantees ACID — every order write is immediately consistent. Best for financial transactions where incorrect data is unacceptable.
  • MongoDB (NoSQL): Designed for distributed systems — chooses A + P. Uses BASE (Basically Available, Soft state, Eventual consistency). A stock count update may take milliseconds to propagate to all replicas — a customer might briefly see outdated inventory.
  • For Orders service: Use PostgreSQL — order creation, payment deduction, and credit checks require strong ACID consistency. A customer must never be double-charged due to eventual consistency lag.
  • For analytics/reporting: MongoDB is acceptable — slight staleness in "total orders today" reporting is acceptable.

Quick-Fire Key Definitions (Memorize These)
DEF
What is CQRS? Define Command Model and Query Model.
▼
CQRS (Command Query Responsibility Segregation) separates an application into two models: the Command Model handles all write operations (create, update, delete) — enforces business rules and emits domain events. The Query Model handles all read operations — uses optimized, denormalized data representations (e.g., Elasticsearch, MongoDB) for fast retrieval. The two models are kept in sync asynchronously via events or CDC.
DEF
What is a Saga? Define compensating transaction.
▼
A Saga is a sequence of local transactions, each of which updates its own database and publishes a message/event to trigger the next step. If any step fails, the saga executes compensating transactions — which are operations that semantically undo previously committed local transactions. Unlike a database ROLLBACK, compensating transactions are application-level operations (e.g., refunding a payment, cancelling an order) because local transactions have already committed and released their locks.
DEF
What is the CAP theorem? What does it mean for microservices?
▼
The CAP theorem states a distributed system can guarantee at most two of: Consistency (all nodes see same data simultaneously), Availability (every request gets a response), and Partition tolerance (system works despite network failures). For microservices: since network partitions are unavoidable (services run on different servers), systems must choose between C or A. NoSQL databases (MongoDB, Cassandra) choose A+P (eventual consistency), while traditional RDBMS choose C+A (no partition tolerance assumed).

What is the ESB in SOA and why is it a problem?
▼
The ESB (Enterprise Service Bus) is the central communication backbone in SOA — all service interactions go through it. It handles routing, transformation, protocol conversion, and orchestration. The problem: the ESB becomes a single point of failure — if it goes down, all services lose the ability to communicate. It also becomes a performance bottleneck — every inter-service call must pass through it. In microservices, the ESB is replaced by lightweight direct communication (REST/gRPC) or decentralized messaging (Kafka), following the principle "smart endpoints, dumb pipes."
