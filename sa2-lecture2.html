Software Development Life Cycle

SDLC هو خريطة الطريق لتطوير أي برنامج — من أول فكرة لحد ما يشتغل ويُدعم

المرحلة الأولى

💡 Concept / Initiation — الفكرة وبدء المشروع

نحدد الحاجة أو المشكلة اللي البرنامج هيحلها. نعمل دراسة أولية.

Scope — حجم المشروع وحدوده Cost Benefit — هل المشروع يستاهل؟ Risk Management Plan — تحديد المخاطر

🧠 سؤال للحفظ: إيه أول خطوة قبل أي برنامج؟ — تعريف الحاجة + دراسة جدوى

المرحلة الثانية

📋 Planning — التخطيط

نعمل خطة شاملة.

جدول زمني الموارد المطلوبة تقسيم المهام

مثال: تحديد موعد بداية المشروع + من المسؤول عن كل مرحلة

المرحلة الثالثة

📝 Requirements — المتطلبات

نحدد وظائف البرنامج بالضبط. نكتب functional requirements: ماذا يجب أن يفعل البرنامج؟ ماذا لا يفعل؟

مثال: برنامج بنك → "يجب أن يسمح بتحويل الأموال بين الحسابات"

المرحلة الرابعة

🎨 Design — التصميم

نترجم المتطلبات إلى تصميم نظامي.

Architecture — شكل البرنامج Data Model — قاعدة البيانات UI/UX — واجهة المستخدم

مثال: نرسم مخطط تدفق البرنامج + نرسم قاعدة البيانات

المرحلة الخامسة

💻 Development — التطوير / البناء

نكتب الكود الحقيقي. نبني كل مكون من مكونات البرنامج.

ملاحظة: هنا يبدأ المطورون يطبقوا التصميم

المرحلة السادسة

🧪 Integration & Test — الدمج والاختبار

ندمج كل المكونات مع بعض ونختبر لو البرنامج يحقق المتطلبات.

Unit Testingاختبار كل مكون لوحده
Integration Testingاختبار المكونات معًا
System Testingاختبار النظام كامل
المرحلة السابعة

🚀 Implement — التنفيذ / التركيب

تثبيت البرنامج في البيئة الفعلية. تدريب المستخدمين على استخدامه.

مثال: تثبيت برنامج البنك في فروع البنك + تدريب الموظفين على استخدامه

المرحلة الثامنة

🔧 Maintenance — الصيانة

دعم البرنامج بعد التنفيذ. إصلاح الأخطاء. تحديث النظام مع الوقت.

Correctiveإصلاح الأخطاء
Adaptiveتعديل حسب التغيرات
Perfectiveتحسين الأداء
Preventiveمنع المشاكل المستقبلية
المرحلة التاسعة

🗄️ Disposition — نهاية دورة البرنامج

لو البرنامج انتهى أو استبدل → نعمل حذف أو أرشفة. نراجع كل الوثائق والدعم.

مراحل SDLC بالترتيب:
Concept → Planning → Requirements → Design → Development → Integration & Test → Implement → Maintenance → Disposition

Software Process Models — أمثلة

من أسوأ نموذج لأشهر نموذج كلاسيكي

⚠️

Code and Fix Model
(أسوأ نموذج تقريبًا 😅)

مفيش تخطيط. مفيش Design. نبدأ نكتب كود على طول.

Code → Test → Fix → Code → Fix → ...

يعني إيه ده عمليًا؟
المبرمج يبدأ يكتب كود مباشرة → يكتشف Bugs → يصلحها → تظهر Bugs تانية → يفضل في loop

✘مفيش Structure
✘صعب جدًا تكبر المشروع
✘Bugs كتير
✘مفيش Documentation

✅ إمتى ممكن يستخدم؟
مشاريع صغيرة جدًا — Prototype سريع

🌊

Waterfall Model
(أشهر نموذج كلاسيكي)

المشروع ماشي خطوة خطوة بشكل خطي — مينفعش ترجع لورا بسهولة.

1. Requirements 2. Design 3. Implementation 4. Verification (Testing) 5. Maintenance

1️⃣ Requirements → نفهم النظام هيعمل إيه
2️⃣ Design → نرسم شكل السيستم
3️⃣ Implementation → نكتب الكود
4️⃣ Verification → نختبر
5️⃣ Maintenance → نصلح ونطور بعد التشغيل

✔سهل الفهم
✔واضح جدًا
✔منظم (Discipline عالي)
✔مناسب لما كل حاجة واضحة من البداية
✘مفيش رجوع (Linear)
✘صعب تغير Requirements
✘كل مرحلة فريق مختلف
✘مفيش Collaboration كفاية
✘لازم تخطط كل حاجة من الأول

💡 أهم جملة تحفظها:
Waterfall ينفع فقط لما Requirements ثابتة وواضحة جدًا

Traditional vs Agile

الطريقة القديمة مقابل الحل الحديث

🔴 Traditional Software Development

بنمشي بخطة ثابتة من الأول للآخر — شبه Waterfall

Less communication — التواصل قليل بين التيم والعميل
العميل يقول المتطلبات مرة واحدة وخلاص
Requirements ثابتة من الأول — بنحدد كل حاجة قبل ما نبدأ
مفيش تغيير بعد كده
بياخد وقت وفلوس أكتر — لأن أي تغيير = كارثة
لازم نرجع نعيد مراحل كتير

💥 المشكلة الكبيرة:
❗ العالم بيتغير، لكن الخطة ثابتة

🟢 Agile — الحل الحديث 🔥

Agile = طريقة تفكير + أسلوب شغل مش مجرد Model

بدل ما نعمل السيستم كله مرة واحدة → نعمله على أجزاء صغيرة (Iterations)

كل Iteration: Plan → Design → Develop → Test → Deploy → Review ↓ نكرر تاني Iteration 1 → Iteration 2 → Iteration 3

🧠 الفكرة العبقرية:
بدل ما تستنى سنة عشان تشوف النتيجة → تشوف جزء شغال كل شوية

🔥 مثال بسيط جدًا
Traditional: تعمل تطبيق كامل مرة واحدة → بعد 6 شهور العميل يقول: مش عايز كده 😅

Agile: تعمل جزء صغير (Login) → تعرضه على العميل → تاخد Feedback → تطور عليه

12 Principles behind Agile Manifesto

دي القوانين الحقيقية اللي Agile ماشي عليها

😊

1️⃣ Customer Satisfaction

أهم هدف = إرضاء العميل. ندي له software شغال بدري وباستمرار.

🔄

2️⃣ Welcome Change

التغيير مرحب بيه حتى لو متأخر. التغيير = ميزة مش مشكلة.

📦

3️⃣ Frequent Delivery

نسلم software كل فترة قصيرة. كل أسبوعين مثلًا.

🤝

4️⃣ Business + Developers Together

العميل + المطورين يشتغلوا سوا يوميًا.

💪

5️⃣ Motivated Team

اختار تيم كويس واديله ثقة. متعملش micromanage.

🗣️

6️⃣ Face-to-Face Communication

أحسن تواصل = مباشر. أسرع + أوضح.

✅

7️⃣ Working Software = Progress

التقدم الحقيقي = برنامج شغال.

⚖️

8️⃣ Sustainable Development

الشغل يكون ثابت مش ضغط مرة واحدة. لا burnout.

💎

9️⃣ Technical Excellence

الكود لازم يكون نظيف. design مهم.

✂️

🔟 Simplicity

اعمل أقل شغل ممكن يحقق الهدف. متعملش over-engineer.

🌀

11️⃣ Self-Organizing Teams

التيم ينظم نفسه. مش لازم manager يقول كل حاجة.

🔍

12️⃣ Continuous Improvement

كل فترة نراجع نفسنا ونحسن. Retrospective.

🔥 طريقة تحفظهم بسهولة

Customer → Delivery → Team → Quality → Simplicity → Improvement
