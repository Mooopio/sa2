2. Interaction Styles: The 2×2 Matrix

Every service-to-service interaction can be classified on two dimensions:

  • Dimension 1 — Cardinality: Is this one client talking to one service, or one client broadcasting to many services?
  • Dimension 2 — Timing: Does the client wait (block) for the response, or does it fire and forget?
TypeStyleDescriptionExample
One-to-OneRequest/Response (Sync)Client sends, waits, blocks until doneHTTP REST API call
One-to-OneAsync Request/ResponseClient sends, does NOT block, reply comes laterMessage queues
One-to-OneOne-Way NotificationClient sends, no reply expected at allFire-and-forget log
One-to-ManyPublish/SubscribeClient publishes to a topic, zero or more services consumeKafka topic
One-to-ManyPublish/Async ResponsesClient publishes, waits a limited time for responsesPrice aggregator

3. Synchronous vs. Asynchronous Communication

The lecture divides microservice communication into two broad types:

Synchronous — Request/Response Pattern

Analogy: Like a phone call. You call someone and you wait, listening, until they answer. You can't do anything else while waiting.

Both the client and the service must be alive and reachable for the entire duration of the exchange. This creates tight coupling. Examples: REST, gRPC, Apache Thrift.

Asynchronous — Messaging Pattern

Analogy: Like sending a text message. You send it and go on with your life. The reply arrives in your inbox whenever the other person responds.

Client and service don't need to be online at the same time. Creates loose coupling. Examples: RabbitMQ, Kafka, ActiveMQ.

4. REST — The Core of Request/Response

What is REST?

REST = Representational State Transfer. It is a set of architectural constraints (not a protocol). RESTful = an API that adheres to those constraints. REST is built entirely on top of HTTP.

Key concept — Resource: A resource is any entity you want to expose to the outside world. Think of it as a "noun" in your API (e.g., /users, /orders/42). HTTP's entire design centers around resources.

REST Advantages

Why developers love REST
  • Simple and familiar — almost every developer knows HTTP
  • Easy to test — use Postman, curl, or even a browser
  • Directly supports request/response — natural fit for client-server interaction
  • Firewall friendly — standard HTTP port 80/443 almost always allowed
  • No intermediate broker needed — simpler system architecture

REST Disadvantages

Where REST falls short
  • Only request/response style — can't do notifications, publish/subscribe natively
  • Reduced availability — both client and service must be running simultaneously
  • Location problem — clients must know the URLs of service instances; needs a service discovery mechanism

5. SOAP — The Old Heavyweight

SOAP (Simple Object Access Protocol) is another approach you can use in microservices, though it's older and less commonly used today.

How SOAP works

SOAP uses HTTP but is more flexible — it can also send messages via JMS (Java Message Service) or SMTP/POP (email). This transport flexibility is a key differentiator from REST.

Message Forwarding & Partial Encryption: A sender can encrypt only certain parts of a SOAP message. The message can then be forwarded through multiple intermediary stations. Each station processes its relevant part, and only the final recipient decrypts and processes the encrypted section. This is enabled by the WS-Security standard.

SOAP Disadvantages

SOAP's limitations
  • Only uses POST — it's essentially an RPC (Remote Procedure Call) mechanism; ignores REST verbs
  • No HATEOAS support — means relationships between services are rigid and hard-coded
  • Tight coupling — interfaces must be fully defined by server and completely known on client upfront

6. HATEOAS — The Secret Sauce of True REST

HATEOAS = Hypermedia as the Engine of Application State

Core idea: Instead of the client hard-coding URLs, the server tells the client what actions are available and where they are — dynamically, in the response itself.

Real-world analogy: Imagine arriving at an unfamiliar city. Without HATEOAS, you need a complete map memorized before you leave home. With HATEOAS, every road sign tells you where the next road leads. You don't need the full map — you just follow the signs.

Without HATEOAS

GET /users/123

{ "id": 123, "name": "Mohamed", "balance": 500 }

The client receives data but has no idea what operations are possible. It must have pre-programmed knowledge of every URL.

With HATEOAS (JSON format)

GET /users/123

{
 "id": 123,
 "name": "Mohamed",
 "balance": 500,
 "links": [
  { "rel": "self", "href": "/users/123" },
  { "rel": "deposit", "href": "/users/123/deposit" },
  { "rel": "withdraw", "href": "/users/123/withdraw" }
 ]
}

The server tells the client exactly what it can do next. If a URL ever changes, the server just updates the response — the client never breaks.

Why HATEOAS Matters for Microservices

  • Loose Coupling: Client doesn't need to hard-code URLs → no tight dependency on server structure
  • Resilience to Change: URLs are generated server-side and sent dynamically → changes don't break clients
  • Without HATEOAS: URL changes in the server require updates in EVERY client app → tight coupling, fragile systems

Common Exam Traps ⚠️

Trap 1: "REST is a protocol." → FALSE. REST is a set of architectural constraints. HTTP is the protocol.
Trap 2: "SOAP only uses HTTP." → FALSE. SOAP can use HTTP, JMS, SMTP/POP.
Trap 3: "HATEOAS is optional for REST." → Technically true, but in exams, HATEOAS is presented as a KEY differentiator that enables loose coupling in REST.
Trap 4: "Synchronous communication doesn't require both parties online." → FALSE. Both must be running for the full duration.
Trap 5: "The Request/Response pattern supports pub/sub." → FALSE. This is explicitly listed as a disadvantage.
