Three-Tier Architecture

أي برنامج كبير بيتقسم 3 طبقات علشان يبقى منظم وسهل التطوير

1

🖥️ Presentation Tier

واجهة المستخدم — User Interface

دي الطبقة اللي المستخدم بيتعامل معاها مباشرة. وظيفتها تعرض البيانات وتاخد input من المستخدم — لكنها مش بتقرر لو الـ input صح أو غلط.

موقع ويب تطبيق موبايل شاشة برنامج

مثال: المستخدم يكتب username + password — الطبقة دي بس بتعرض وبتستقبل.

2

🧠 Logic Tier

Business Logic — مخ البرنامج

دي مخ البرنامج. هي اللي بتفكر وتنفذ القوانين. الـ Logic Layer هو الوحيد اللي يتكلم مع الداتابيز.

هل كلمة السر صحيحة؟ هل المستخدم Admin؟ هل يقدر يشوف الصفحة؟

مثال Login: بتتحقق من البيانات → بتقارنها بالداتابيز → بتقرر يدخل أو لا

3

🗄️ Data Tier

Database — طبقة تخزين البيانات

دي طبقة تخزين البيانات. المستخدم ميقدرش يوصل للداتابيز مباشرة — بس Logic Layer يقدر.

Users Passwords Posts Products
MySQL PostgreSQL MongoDB

🛒 مثال عملي — شراء منتج من موقع

1️⃣ المستخدم يضغط Buy
↓
2️⃣ الواجهة تبعت الطلب للسيرفر
↓
3️⃣ السيرفر يتأكد من الفلوس والكمية
↓
4️⃣ يسجل الطلب في الداتابيز ✅
ثانيًا وثالثًا

Monolithic vs Microservices

أسلوبان مختلفان كليًا في بناء البرامج الكبيرة

🧱

Monolithic Architecture

كلمة Monolithic معناها: كل البرنامج قطعة واحدة

يعني الـ UI والـ Business Logic والـ Database Access كلهم داخل برنامج واحد كبير.

برنامج واحد فيه: ├─ تسجيل دخول ├─ منتجات ├─ دفع ├─ رسائل └─ كل حاجة
✔سهل في البداية
✔أسهل في التطوير الصغير
✘لو البرنامج كبر يبقى صعب
✘أي مشكلة توقف السيستم كله
✘صعب يشتغل عليه فرق كبيرة
🔬

Microservices Architecture

بدل برنامج كبير، نعمله برامج صغيرة كتير. كل برنامج مسؤول عن حاجة واحدة. كل service له كوده وداتابيزه ويشتغل لوحده.

👤
User Service
User DB
📦
Order Service
Order DB
💳
Payment Service
Payment DB
✔Scalable
✔سهل التطوير والتعديل
✘أكثر تعقيدًا في البداية
الخاصية Monolithic Microservices
البنيةبرنامج واحد كبيربرامج صغيرة كتير
الكودكل الكود مع بعضكل خدمة منفصلة
التعديلصعب التعديلسهل التطوير

Microservices Design Patterns

patterns = طرق جاهزة لحل مشاكل التصميم

🏢 Decompose by Business Capability

نقسم حسب وظائف البزنس

User Service Order Service Payment Service

🗂️ Decompose by Subdomain

نقسم حسب جزء من النظام

Product Domain Order Domain Shipping Domain Payment Domain

🐍 Strangler Pattern

طريقة لتحويل Monolithic → Microservices بالتدريج. نبدأ ناخد جزء من السيستم القديم ونحوله service، لحد ما السيستم القديم يختفي.

🚪 API Gateway

هو باب واحد لكل السيستم. المستخدم بدل ما يكلم كل service — يكلم API Gateway والـ Gateway يوزع الطلب.

User → API Gateway ├→ User Service ├→ Order Service └→ Payment Service

📦 Aggregator Pattern

service تجمع البيانات من عدة services.

صفحة Profile تحتاج: - user service - order service - payment service ↓ Aggregator يجمعهم ويرجع رد واحد

🖥️ Client-Side UI Composition

الـ Frontend نفسه يجمع البيانات من services مختلفة.

🗃️ Database per Service

كل Service لها database خاصة. ده الأفضل ✅

User Service → User DB Order Service → Order DB

⚠️ Shared Database

كل services تستخدم database واحدة. ده أسهل لكن خطر.

↕️ CQRS Pattern

نقسم العمليات:

Read → قراءة لوحدها Write → كتابة لوحدها

🔄 Saga Pattern

لحل مشكلة transactions بين services. لو واحدة فشلت، السيستم يرجع كل العمليات.

شراء منتج يحتاج: ├─ خصم فلوس ├─ حجز المنتج └─ تسجيل الطلب

📋 Log Aggregation

نجمع logs من كل services في مكان واحد علشان نراقبهم.

📈 Performance Metrics

قياس الأداء.

- latency - request count

🔍 Distributed Tracing

نعرف الطلب مر على أي services.

💓 Health Check

نراقب: هل service شغالة أم لا.

⚙️ Externalized Configuration

نخلي الإعدادات خارج الكود.

مثال: config server

🔍 Service Discovery

service تعرف عنوان services الأخرى.

- Eureka - Consul

⚡ Circuit Breaker

لو service تعبانة، نوقف الطلبات مؤقتًا علشان نحمي السيستم.

Communication Patterns

أنماط التواصل بين الخدمات — الميكروسيرفيسز لازم تتكلم مع بعض

Synchronous

1️⃣ Request / Response Pattern

الخدمة A تطلب شيء من الخدمة B، والخدمة B ترد مباشرة. شبه لما المستخدم يضغط على زر ويستنى الرد فورًا. الخدمة A مستنية الرد قبل ما تكمل شغلها.

User Service → Order Service "اعطني الطلبات" Order Service ← يرد بالطلبات
✅ مميزات
  • ✔ سهل الفهم
  • ✔ الرد فورًا
❌ عيوب
  • ✘ لو B بطيئة، A تتأخر
  • ✘ مش scalable
Asynchronous

2️⃣ Messaging Pattern

الخدمات تبعت رسائل عبر نظام وسيط (Queue أو Broker). كل خدمة ممكن تستقبل الرسائل وقت ما تحب.

Order Service → [Kafka/RabbitMQ] ↓ Payment Service ← تلتقط Shipping Service ← تلتقط
Kafka RabbitMQ
✅ مميزات
  • ✔ أكثر scalable
  • ✔ الخدمات مستقلة
  • ✔ لو تعطلت، تنتظر
❌ عيوب
  • ✘ صعب debug
  • ✘ في بعض التأخير
Asynchronous

3️⃣ Event Driven Pattern

كل خدمة تراقب الأحداث في السيستم وتتصرف على أساسها. الخدمات لا تبعت رسائل مباشرة لبعضها، بل تبعت أحداث عامّة.

حدث: "User Created" ↓ Notification Service → إيميل ترحيبي Analytics Service → تسجيل بيانات Welcome Service → شاشة ترحيب
✅ مميزات
  • ✔ سيستم مرن جدًا
  • ✔ قابل للتوسع الكبير
  • ✔ لا تعتمد مباشرة
❌ عيوب
  • ✘ معقد شويه
  • ✘ لازم مراقبة جيدة

ملخص بسيط للحفظ

النمط Sync/Async وصف سريع
Request/Response Synchronous الخدمة A تطلب، الخدمة B ترد فورًا
Messaging Asynchronous الرسائل تنتقل عبر Broker، كل خدمة تلتقط وقت ما تحب
Event Driven Asynchronous كل خدمة تراقب أحداث وتتصرف عليها، لا تبعت رسائل مباشرة

Deployment Patterns

أنماط النشر للخدمات — إزاي ننشرهم على السيرفرات بطريقة صحيحة

🏠

Multiple Service Instances per Host

ننشر أكثر من خدمة على نفس السيرفر. يعني السيرفر الواحد فيه نسخ متعددة من services مختلفة.

Server 1 ├─ User Service ├─ Order Service └─ Payment Service
✔أقل تكلفة، استغلال أفضل
✘لو السيرفر وقع — كل شيء وقع
🏢

Service Instance per Host

كل خدمة على سيرفر منفصل.

Server 1 → User Service Server 2 → Order Service Server 3 → Payment Service
✔عزل واستقرار أفضل
✘أغلى — سيرفر لكل خدمة
☁️

Serverless Deployment

الخدمة تعمل بدون سيرفر تقليدي. الكود بيشتغل على cloud provider حسب الحاجة. تدفع بس على حسب الاستخدام.

AWS Lambda Azure Functions
✔ما تحتاجش تدير السيرفرات
✔scale تلقائي
✘مش مناسب لكل الخدمات
⚙️

Service Deployment Platform

بيئة أو منصة خاصة لإدارة نشر الخدمات. المنصة تدير كل حاجة: نشر، مراقبة، scaling، واسترجاع الخدمة لو وقعت.

Kubernetes Docker Swarm AWS ECS
✔تنظيم وscaling متقدم
✘setup معقد، خبرة مطلوبة

ملخص سريع للحفظ

النمط وصف سريع مميزات عيوب
Multiple Service Instances per Host أكثر من خدمة على سيرفر واحد رخيص، سهل لو السيرفر وقع، كل الخدمات توقف
Service Instance per Host كل خدمة على سيرفر منفصل عزل واستقرار أغلى، إدارة أصعب
Serverless Deployment بدون سيرفر، cloud functions scale تلقائي، إدارة سهلة مش مناسب لكل شيء، runtime محدود
Service Deployment Platform منصة لإدارة نشر الخدمات تنظيم، مراقبة، scalable setup معقد، خبرة مطلوبة
