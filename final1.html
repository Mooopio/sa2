Communication Patterns Overview

Synchronous (Request/Response)

  • Client blocks waiting for response
  • Both client & service must be available
  • Tight runtime coupling
  • Protocols: REST, gRPC, Apache Thrift

Asynchronous (Messaging)

  • Client does NOT block
  • Messages buffered by broker
  • Loose runtime coupling
  • Brokers: RabbitMQ, Kafka, ActiveMQ

REST (Representational State Transfer)

  • Built entirely on top of HTTP
  • Key abstraction = Resource (anything exposed to the world)
  • REST = constraints; RESTful = API adhering to those constraints

✅ Advantages

  • Simple & familiar
  • Testable via Postman / curl
  • Firewall friendly (HTTP)
  • No intermediate broker
  • Direct request/response support

❌ Disadvantages

  • Only supports request/response style
  • Both ends must be running (reduced availability)
  • Clients must know service URLs (service discovery needed)

HATEOAS

Hypermedia As The Engine Of Application State — a REST component where server returns URLs dynamically in responses so clients don't need to hard-code them.

Key Benefit Enables loose coupling. URLs are generated server-side and sent in JSON/XML responses. No hard-coded endpoints = no breakage when URLs change.
Without HATEOAS API endpoints must be hard-coded on client. Any endpoint change breaks all client apps → tight coupling.

Messaging Pattern (Asynchronous)

Message Structure

  • Header: name-value metadata pairs
  • Body: actual data (text or binary)

Message Types

TypeDescriptionExample
DocumentGeneric data; receiver decides how to use itReply to a command
CommandSpecifies operation + parameters (like RPC)"PlaceOrder" message
EventSomething notable happened (domain event)OrderPlaced, CustomerUpdated

Message Formats

Text (JSON / XML)

  • Human-readable, self-describing
  • ❌ Contains attribute names = overhead
  • ❌ Parsing overhead on large messages

Binary (Protocol Buffers / Avro)

  • More compact, saves bandwidth
  • Better for performance-critical systems
  • Not human-readable

Channel Types

  • Point-to-Point: Delivers to exactly ONE consumer → used for one-to-one interactions
  • Publish-Subscribe: Delivers to ALL attached consumers → used for one-to-many interactions

Implementing Interaction Styles with Messaging

StyleHow it works
Async Req/ResponseClient sends command msg with reply-channel header. Service replies with correlation ID. Client matches IDs.
One-way NotificationClient sends to point-to-point channel. Service processes. No reply.
Publish/SubscribePublisher sends to pub-sub channel. All subscribed services receive it.
Publish/Async ResponsesClient publishes with reply-channel header. Consumers reply with correlation ID. Client collects.

Data Management Patterns

Database per Service

  • Each service has its own private database (3 options: private tables, schema-per-service, database-server-per-service)
  • ✅ Loose coupling, best-fit DB type per service
  • ❌ Cross-service transactions hard (no distributed ACID), joins challenging

Shared Database

  • Multiple services share one DB
  • ✅ Simple ACID transactions, easier to operate
  • ❌ Development-time coupling, runtime coupling, locking issues, one-size-fits-all DB

CAP Theorem

⚠️ Critical Concept A distributed system can provide at most 2 of 3: Consistency, Availability, Partition Tolerance. In RDBMS → C+A (no P). In NoSQL → want P, so choose C or A.

ACID vs BASE

ACID (SQL)

  • Atomicity – all or nothing
  • Consistency – always valid state
  • Isolation – transactions don't interfere
  • Durability – committed = permanent

BASE (NoSQL)

  • Basically Available – partial failures ok
  • Soft state – may change without input
  • Eventual Consistency – consistent eventually

NoSQL Database Types

Graph DB Key Concepts

  • Nodes = Entities (like rows in RDBMS)
  • Edges (Relationships) = Connections (like foreign keys)
  • Labels = Group similar nodes
  • Properties = Key/value pairs on nodes or relationships
  • Relationships are first-class citizens — stored explicitly, not calculated at query time

Other Data Patterns (Brief)

  • SAGA: Sequence of local transactions for cross-service data consistency
  • CQRS: Materialized views for efficient queries
  • Event Sourcing: Persist aggregates as sequence of events
  • Domain Event: Publish an event whenever data changes

Q: Compare synchronous and asynchronous communication in microservices.
1.Definition: Synchronous communication (e.g., REST) requires the client to wait for a response. The client may block during this time. Asynchronous communication (messaging) allows the client to continue without waiting.
2.Coupling: Synchronous creates tight runtime coupling — both must be available simultaneously. Asynchronous achieves loose coupling via a message broker that buffers messages.
3.Availability: Synchronous reduces availability (both ends must be up). Asynchronous improves availability (broker holds messages until consumer is ready).
4.Interaction styles: REST only supports request/response. Messaging supports all styles: async req/response, one-way, publish/subscribe, publish/async responses.
5.Examples: Synchronous: REST, gRPC, Apache Thrift. Asynchronous: RabbitMQ, Apache Kafka, ActiveMQ.
Q: Explain the CAP theorem and its impact on database design in microservices.
1.Definition: The CAP Theorem states that a distributed system can only simultaneously guarantee 2 of 3 properties: Consistency, Availability, and Partition Tolerance.
2.RDBMS: Traditional relational databases have no Partition Tolerance → they achieve Consistency + Availability (C+A) using ACID guarantees.
3.NoSQL: NoSQL databases are designed with Partition Tolerance (P). They must then choose either Consistency (drop A) or Availability (drop C), leading to BASE semantics instead of ACID.
4.Impact: The Database per Service pattern means cross-service transactions cannot use distributed ACID transactions (CAP makes this impractical), requiring patterns like SAGA instead.
Q: Compare Database per Service vs Shared Database pattern.
1.Database per Service: Each microservice has its own private database accessible only via its API. Can use private tables, schema-per-service, or separate DB server.
2.DB/Service Benefits: Loose coupling (changes to one service's DB don't affect others), flexibility (each service can use the best-fit database type e.g., Neo4j for graph data).
3.DB/Service Drawbacks: Cross-service transactions are complex (no distributed ACID), joins across databases are challenging, managing multiple DB types adds complexity.
4.Shared Database Benefits: Simple ACID transactions, single DB easier to operate.
5.Shared Database Drawbacks: Development-time coupling (schema changes require coordination), runtime coupling (long transactions can lock tables used by other services), single DB may not fit all service needs.
Q: What is HATEOAS and why is it important in RESTful microservices?
1.Definition: HATEOAS (Hypermedia As The Engine Of Application State) is a REST component where server responses include dynamically generated URLs (hypermedia links) that guide client actions.
2.Problem solved: Without HATEOAS, clients must hard-code API endpoints. Any endpoint change breaks client applications — this is tight coupling.
3.Solution: With HATEOAS, URLs are generated server-side and returned in JSON/XML responses. Clients always use the latest URLs from the server → loose coupling.
4.Formats: HATEOAS links can be returned in JSON or XML format, containing the rel (relationship) and href (URL) for available actions.
