1.1 Two-Phase Commit (2PC)

🎯 What is 2PC? — The Core Concept

Imagine you're organizing a group dinner. Before the chef starts cooking, you call every guest and ask: "Can you make it tonight?" Only if everyone says YES does the dinner proceed. If even one person says NO, you cancel for everyone. That's exactly how 2PC works — it's an all-or-nothing agreement mechanism across distributed databases.

Formal definition: 2PC is a distributed algorithm that guarantees atomicity — either all participating nodes commit a transaction, or none of them do.

Atomicity Coordinator Participants All-or-Nothing

The Two Roles

🎮 Coordinator

  • Starts the transaction
  • Sends PREPARE messages to all participants
  • Collects votes from participants
  • Broadcasts final COMMIT or ABORT decision

🖥 Participants

  • Execute local transaction steps
  • Write actions to recovery log
  • Vote COMMIT if ready
  • Vote ABORT if any problem exists

Phase-by-Phase Walkthrough

Phase 1 — Prepare (Voting)
The coordinator sends a PREPARE message ("Are you ready to commit?"). Each participant executes the transaction up to the commit point, writes to its recovery log, then replies VOTE-COMMIT (ready) or VOTE-ABORT (problem detected).
Phase 2 — Commit / Abort
If all participants voted COMMIT → coordinator sends COMMIT to all. If any participant voted ABORT → coordinator sends ABORT to all. Each participant finalizes and releases held locks, then sends ACK.
Coordinator Participant 1 Participant 2 ─── PHASE 1: PREPARE ─────────────────────────── ──► PREPARE ──────────────────────────────────► ◄── VOTE-COMMIT ◄────────────────────────────── ─── PHASE 2: COMMIT ──────────────────────────── ──► COMMIT ───────────────────────────────────► ◄── ACK ◄──────────────────────────────────────

Why 2PC Fails for Microservices — 6 Critical Problems

  1. Blocking Protocol: Participants lock resources until ALL parties respond. If the coordinator crashes mid-commit, every participant freezes forever — they can't commit or rollback independently.
  2. Single Point of Failure: If the coordinator dies during Phase 2, participants are in limbo — they don't know whether to commit or rollback, and they stay blocked indefinitely.
  3. Tight Coupling: Microservices must be independent. 2PC creates hard cross-service dependencies, violating the fundamental principle of microservices architecture.
  4. Poor Fault Tolerance: Network failures and node crashes are extremely common in distributed systems. 2PC cannot handle these scenarios gracefully.
  5. Performance Bottleneck: The additional round-trips and locking add significant latency, making it unsuitable for high-throughput microservices.
  6. No Native Broker Support: Kafka, RabbitMQ, and REST (the standard tools of microservices) simply do not support distributed 2PC.

1.2 Three-Phase Commit (3PC)

🔧 What is 3PC? — Fixing 2PC's Biggest Problem

3PC adds a middle phase — PreCommit — as a "warning shot" before the final commit. Think of it like a group project where you first check everyone's availability (CanCommit?), then tell everyone "we're definitely doing this, get ready" (PreCommit), and finally say "GO!" (DoCommit). The key insight: if the project manager (coordinator) disappears after the "get ready" announcement, everyone already knows a commit is coming and can proceed independently.

Introduced by Dale Skeen in 1982 to eliminate indefinite blocking caused by coordinator failure in 2PC.

The 2PC Problem 3PC Solves
In 2PC: if both the coordinator AND a participant fail during the Commit phase, a new coordinator cannot determine whether a commit occurred — all remaining participants block indefinitely. 3PC introduces a "Prepared-to-Commit" state that allows any new coordinator to infer the correct decision.

The Three Phases in Detail

Phase 1 — CanCommit (Vote)
Coordinator sends CanCommit? to all participants. Each replies YES or NO. If ANY says NO → coordinator sends immediate ABORT to all.
Phase 2 — PreCommit (Prepare)
If all voted YES → coordinator sends PreCommit. Participants write to recovery log, enter "prepared" state (not yet committed), send ACK. KEY: All participants now KNOW a commit is coming — coordinator can safely fail here.
Phase 3 — DoCommit (Commit)
Coordinator sends DoCommit. Participants make changes permanent, release locks, send final ACK. If coordinator fails HERE, participants can safely commit independently (they're all in PreCommit state).
Coordinator Participant 1 Participant 2 ─── PHASE 1: CanCommit ────────────────────────── ──► CanCommit? ───────────────────────────────► ◄── YES ◄────────────────────────────────────── ─── PHASE 2: PreCommit ────────────────────────── ──► PreCommit ────────────────────────────────► ◄── ACK ◄────────────────────────────────────── ─── PHASE 3: DoCommit ─────────────────────────── ──► DoCommit ─────────────────────────────────► ◄── ACK ◄──────────────────────────────────────
Critical Limitation of 3PC
3PC assumes bounded network delays. In real-world systems with network partitions (split-brain scenarios), 3PC can violate atomicity — different partitions may make conflicting commit decisions. This is why it's rarely used in production.

1.4 Saga Pattern — The Right Answer for Microservices

⚡ Context: The Problem Saga Solves

In microservices, each service owns its own database (the Database Per Service pattern). When a business transaction spans multiple services — like placing an order that must check inventory AND charge a customer — you cannot use a single ACID transaction. 2PC and 3PC are not viable.

Classic Example: An e-commerce order must not exceed a customer's credit limit. Orders and Customers live in separate databases — a simple local ACID transaction is impossible.

Saga Definition
Implement each business transaction spanning multiple services as a saga — a sequence of local transactions. Each local transaction updates the database and publishes an event. If a step fails, compensating transactions undo all previously completed steps.

Two Coordination Approaches

🎵 Approach 1: Choreography (Event-Driven)

Analogy: A relay race. Each runner (service) runs their leg, then passes the baton (event) to the next runner. No coach (central coordinator) tells each runner what to do — they react to receiving the baton automatically.

  1. Order Service creates order → publishes OrderCreatedEvent
  2. Payment Service listens → charges customer → publishes PaymentCompletedEvent
  3. Inventory Service listens → reserves stock → publishes InventoryReservedEvent
Failure & Compensation
If Payment fails → publishes PaymentFailedEvent → Order Service listens → cancels order (compensating action). If Inventory fails → publishes InventoryFailedEvent → Payment Service listens → issues refund.

✔ Pros

  • Loosely coupled services
  • No single point of failure
  • Natural for event-driven architecture
  • Works great with Kafka/RabbitMQ
  • Simpler for small workflows

✗ Cons

  • Hard to understand overall flow
  • Complex failure handling
  • No central monitoring point
  • Event ordering challenges
  • Coordination effort grows with complexity
🎯 Approach 2: Orchestration (Centralized)

Analogy: A conductor leading an orchestra. The conductor (orchestrator) tells each musician (service) exactly when and what to play. The orchestra members don't know about each other — only the conductor knows the full score.

A central CreateOrderSaga orchestrator manages the entire flow:

  1. Orchestrator → Order Service: createOrder() → sets state = PENDING
  2. Orchestrator → Payment Service: reserveCredit() → charge customer
  3. Orchestrator → Inventory Service: reserveInventory() → reserve stock → state = APPROVED
On Failure
If creditLimitExceeded → Orchestrator sends rejectOrder() to Order Service → state = REJECTED. (Inventory not yet reached → no compensation needed there.)

✔ Pros

  • Centralized control & visibility
  • Clear business logic flow
  • Better error handling & retries
  • Easy monitoring & observability
  • Consistent flow execution

✗ Cons

  • Tight coupling with orchestrator
  • Single point of failure
  • Scalability constraints
  • Complexity accumulates ("god service")
  • Less reactive / asynchronous

Saga: Rolling Back & API Design

⚠ Compensating Transactions ≠ ROLLBACK
Unlike a simple SQL ROLLBACK TRANSACTION, rolling back a saga requires compensating transactions — application-level logic that semantically reverses each committed step. Each completed step needs a corresponding compensating transaction written by the developer.

Saga Orchestrator — How It Works

A saga orchestrator is a persistent object that tracks saga state and invokes participants.

On Create

  • Invokes the first saga participant
  • Persists state in the database
  • Determines which participant to invoke next

On Reply

  • Loads current state from database
  • Determines the next participant
  • Invokes that participant
  • Updates & persists its state

Smart Summary

High-yield bullet points · Hierarchical · Exam-focused

🔵 Two-Phase Commit (2PC)

  • Purpose: Guarantee atomicity across distributed nodes — all-or-nothing commit
  • Roles: Coordinator (controls) + Participants (execute & vote)
  • Phase 1 (Prepare): Coordinator sends PREPARE → participants vote COMMIT/ABORT
  • Phase 2 (Commit/Abort): All COMMIT → send COMMIT; any ABORT → send ABORT to all
  • Key guarantee: Atomicity ✔ · Strong consistency ✔
  • Key weakness: Blocking if coordinator fails ✗ · Resources locked until resolved ✗
  • Why NOT for microservices: Blocking · Single-point-of-failure · Tight coupling · No broker support · Performance bottleneck · Poor fault tolerance

🟡 Three-Phase Commit (3PC)

  • Created by: Dale Skeen, 1982 — extension of 2PC
  • Core fix: Adds PreCommit phase to eliminate indefinite blocking
  • Phase 1 (CanCommit): Vote YES/NO — any NO → immediate ABORT
  • Phase 2 (PreCommit): All say YES → enter "prepared" state — commit is guaranteed
  • Phase 3 (DoCommit): Make changes permanent, release locks
  • Key advantage: Non-blocking recovery — participants can decide independently after PreCommit
  • Key limitation: Assumes bounded delays · NOT partition-safe · 3 RTTs · Complex · Rarely used in production

🟢 Saga Pattern

  • Context: Database-per-service pattern — cross-service transactions needed
  • Definition: Sequence of local transactions, each publishing events; compensating transactions on failure
  • Consistency model: Eventual consistency (not strong/ACID)
  • Rollback mechanism: Compensating transactions (NOT database ROLLBACK)
  • Approach 1 — Choreography: Event-driven; each service reacts to events; no central controller; loosely coupled; harder to debug
  • Approach 2 — Orchestration: Central orchestrator commands each service; clear flow; single point of failure risk; easier to monitor
  • Orchestrator: Persistent object — tracks state, persists to DB, invokes participants
  • API recommendation: Asynchronous (respond immediately, client polls for outcome)

⚖️ 2PC vs 3PC vs Saga — Quick Reference

  • 2PC: 2 phases · blocking · simple · used in RDBMS/XA · NOT for microservices
  • 3PC: 3 phases · non-blocking · complex · partition-unsafe · academic/rare
  • Saga: N local transactions · eventual consistency · compensating rollback · THE microservices solution
  • Choreography vs Orchestration: No central brain vs Central brain · Events vs Commands · Decoupled vs Controlled

⚠️ Common Exam Traps & Mistakes

Trap 1: "3PC is always better than 2PC"
❌ FALSE. 3PC is non-blocking, but it's more complex, has higher latency (3 vs 2 RTTs), and is NOT partition-safe. In real production systems, 3PC is rarely used. The better answer for microservices is the Saga Pattern.
Trap 2: "Saga uses ROLLBACK to undo transactions"
❌ FALSE. Saga uses compensating transactions — application-level logic written by the developer. There is NO database ROLLBACK. Each committed local transaction has a corresponding compensating transaction that semantically reverses it.
Trap 3: "Choreography has a central coordinator"
❌ FALSE. Choreography is the decentralized approach — no central coordinator. Services react autonomously to events. The Orchestration approach has a central coordinator (the orchestrator).
Trap 4: "2PC guarantees consistency in microservices"
❌ MISLEADING. While 2PC guarantees atomicity, it's not suitable for microservices because of blocking, tight coupling, no broker support, and poor fault tolerance. The Saga Pattern provides eventual consistency which is the correct trade-off for microservices.
Trap 5: "3PC solves network partitions"
❌ FALSE. 3PC can actually violate atomicity during network partitions (split-brain scenario). It assumes bounded network delays, which don't exist in real-world distributed systems. 3PC reduces blocking but does NOT solve the partition problem.
Trap 6: "Orchestration is better than Choreography"
❌ NEITHER IS UNIVERSALLY BETTER. Orchestration offers better control and monitoring. Choreography offers better decoupling and resilience. Choice depends on system complexity, team structure, and requirements.
