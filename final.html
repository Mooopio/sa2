A · Software Process Models

A software process model = abstract description of a software development process. All processes share 4 core activities:

  • Specification – what the system should do
  • Design & Implementation – how to build it
  • Validation – does it match customer needs?
  • Evolution – changing in response to needs

Waterfall

  • Linear phases: Requirements → Design → Implementation → Testing → Deployment
  • Benefits: Discipline, simple, easy to manage, good for stable requirements (mass production)
  • Drawbacks: No going back, each phase owned by different team, little collaboration, rigid scope
  • Code & Fix: worst model — no design, just code then fix bugs late
Waterfall Stats: 86% of projects fail/challenged; <30% complete as planned; 30% cancelled before they begin. (Standish Group)

Triple Constraints

Scope is Fixed; Time and Cost are Flexible (in traditional Waterfall thinking)

C · DevOps

DevOps bridges the gap between Developers and Operations. It evolved from Waterfall → Agile → DevOps to resolve the last-mile deployment gap.

Why DevOps? Waterfall was rigid; Agile fixed dev-customer gap but Operations still slowed releases; DevOps unifies Dev + Ops into one team.

DevOps Overview

  • One integrated team → better communication & collaboration
  • Automate testing, infrastructure, workflows
  • Continuous performance measurement
  • Code written in small chunks; deployments in days/hours not weeks
  • Dev environment = Production environment
  • Infrastructure as Code (Configuration Management)

  • Breaking Silos — cross-team collaboration
  • Continuous Improvement — reduced feedback cycle → higher customer satisfaction

  • Creativity & Innovation — trust culture
  • Performance-oriented Culture — reduced turnover

CI/CD

  • CI (Continuous Integration): Code changes merged regularly and frequently into shared repo; small changes at high frequency → catch problems early → stable, up-to-date codebase
  • CD (Continuous Deployment): Automatically deploy merged changes to staging/production
  • CI/CD Pipeline: Build → Test → Deploy changes automatically; alerts on problems
  • Tools: Git (version control), Jenkins (automation), Ansible (deployment)

D · Monolithic vs Microservices vs SOA

Monolithic Architecture

  • Single deployable unit; all modules in one process; hexagonal architecture (business logic + adapters)
  • Early Advantages: Simple to develop, easy radical changes, straightforward test & deploy, easy to scale (basic load balancer)
  • Monolithic Hell Disadvantages: Slow development, no individual scaling, long commit-to-deploy cycle, locked-in tech stack, unreliable (no fault isolation), difficult testing

Microservices

  • Independently deployable modules; each solves one specific problem (single responsibility)
  • Evolution of SOA; loosely coupled; each owns its own database; no shared tech stack required
  • Principles: Modelled around business domain, Culture of Automation (CI/CD), Hide implementation details, Decentralization, Deploy Independently, Failure Isolation, Highly Observable

SOA — Service Oriented Architecture

  • Enterprise-wide approach; reusable services across platforms/languages
  • Roles: Service Provider (publishes in registry) + Service Consumer (finds & binds service)
  • ESB (Enterprise Service Bus): Common communication protocol; routes messages, transforms data, handles protocols; can be single point of failure
  • Registry: Controlled access to service metadata; governs SOA
  • 4 Service Types: Functional (business), Enterprise, Application, Infrastructure
  • Service = Interface + Contract + Implementation
  • Advantages: Reusability, easy maintenance, platform-independent, availability, reliability, scalability, faster time to market
  • Disadvantage: ESB = bottleneck & single point of failure

F · Data Management Patterns

1 — Two-Phase Commit (2PC)

  • Distributed algorithm ensuring all nodes agree to commit or abort
  • Roles: Coordinator (initiates, collects votes, sends decision) + Participants (execute, vote, ACK)
  • Phase 1 — Prepare: Coordinator sends PREPARE → Participants execute up to commit point, write to recovery log, vote COMMIT or ABORT
  • Phase 2 — Commit/Abort: All COMMIT votes → send COMMIT; any ABORT vote → send ABORT; Participants finalize & release locks
  • Guarantees: Atomicity + Strong consistency
Why 2PC fails in Microservices: Blocking protocol (locks resources), Poor fault tolerance, Single point of failure (coordinator crash), Performance bottleneck, Tight coupling, No native broker support (Kafka/RabbitMQ don't support 2PC)

2 — Three-Phase Commit (3PC)

  • Extension of 2PC by Dale Skeen (1982); adds intermediate PreCommit phase to prevent indefinite blocking
  • Phase 1 — CanCommit: Coordinator asks "ready to commit?" → YES/NO
  • Phase 2 — PreCommit: If all YES → coordinator sends PreCommit → participants write to log, enter "prepared" state, send ACK. KEY: all now KNOW a commit is coming
  • Phase 3 — DoCommit: Coordinator sends DoCommit → participants make permanent, release locks, send ACK. If coordinator fails here, participants can commit independently
  • Solves: Indefinite blocking when coordinator + participant both fail during commit
  • Limitations: Higher latency (3 RTTs), complex, NOT partition-safe (split-brain), assumes bounded delays, rarely used in production

3 — Saga Pattern

A saga = sequence of local transactions. Each local transaction updates its own database and publishes an event/message to trigger the next step. If a step fails → compensating transactions undo all prior completed steps.

ChoreographyOrchestration
ControlDistributed — no central coordinatorCentralized — saga orchestrator
CommunicationEvents via broker (Kafka/RabbitMQ)Direct commands to services
CouplingLoosely coupledServices depend on orchestrator
Failure HandlingEach service handles own compensationOrchestrator sends compensating commands
DebuggingHard — flow spread across servicesEasy — single place to track
SPOFNoneOrchestrator crash halts saga
VisibilityLowHigh

Choreography Example: Order Service emits OrderCreatedEvent → Payment Service listens, charges, emits PaymentCompletedEvent → Inventory Service listens, reserves stock. If Payment fails → emits PaymentFailedEvent → Order Service cancels order.

Orchestration Example: CreateOrderSaga Orchestrator → tells Order Service (createOrder) → tells Payment Service (reserveCredit) → tells Inventory Service (reserveInventory). If credit fails → orchestrator sends rejectOrder to Order Service.

Rolling Back Sagas: No simple ROLLBACK. Developer writes compensating transactions that semantically reverse previous steps. APIs should prefer asynchronous response (better availability) — send response immediately after creating saga, client polls/gets notified on completion.

4 — CQRS Pattern

Command Query Responsibility Segregation — separates read operations (queries) from write operations (commands) into distinct models and often separate databases.

  • Command Model: Handles state changes (CreateOrder, UpdateCustomer); command handlers validate inputs, enforce business rules, emit domain events
  • Query Model: Read-only, optimized for performance; uses Elasticsearch or denormalized SQL; does NOT modify state
  • Eventual Consistency: Write and read models are decoupled; changes propagate asynchronously via domain events or CDC
  • Typical Setup: Relational DB for writes (CRUD) + NoSQL for reads (queries)

Other Data Patterns (Summary)

  • Database per Service: Each microservice owns its private DB — no sharing
  • Shared Database: Multiple services share one DB (anti-pattern for pure microservices)
  • Event Sourcing: Persist aggregates as sequence of events (full audit trail)
  • Domain Event: Publish event whenever data changes

Analogies

Saga Pattern = Relay Race: Each runner (service) runs their leg (local transaction). If a runner drops the baton (fails), previous runners reverse (compensating transactions) — the team doesn't finish wrong, they restart correctly.
2PC = Group Vote to Sign a Contract: Everyone votes YES/NO (Phase 1). If all YES → everyone signs (Phase 2). If coordinator disappears after Phase 1 — everyone is stuck waiting forever.
3PC = Building Permits: First ask "Can you build?" (CanCommit) → get approval "Prepare to build" (PreCommit) → then "Build now" (DoCommit). Even if the inspector vanishes after PreCommit, you know it was approved — you can safely build.
ESB = Office Receptionist: All calls (messages) go through one receptionist (ESB). Great for routing — but if receptionist is sick (SPOF), the whole office stops.
CQRS = Restaurant Kitchen vs Menu: Kitchen (Command model) changes food. Menu display (Query model) shows what's available. They're separate — you read the menu (query) and give orders (command) through different channels.
Choreography vs Orchestration = Jazz vs Orchestra: Jazz (choreography) — musicians react to each other freely; no conductor. Orchestra (orchestration) — conductor tells everyone when and what to play.
DevOps = Factory Assembly Line: Dev = designs the car; Ops = runs the factory. DevOps = same team designs AND runs the factory, so improvements are instant and continuous.

Q: Compare Choreography and Orchestration in the Saga Pattern. [8 marks]

The Saga Pattern implements cross-service transactions as a sequence of local transactions with compensating transactions for rollback. There are two coordination approaches:

Choreography (Event-Driven): In this approach, each service reacts autonomously to domain events published to a message broker (e.g., Kafka/RabbitMQ). There is no central coordinator. Services are loosely coupled — they only know about events they publish or consume. Example: Order Service emits OrderCreatedEvent → Payment Service listens and charges customer → emits PaymentCompletedEvent. If payment fails, it emits PaymentFailedEvent and Order Service cancels the order. Advantages: no single point of failure, promotes service autonomy. Disadvantages: flow is hard to trace, complex failure handling.

Orchestration (Centralized): A central saga orchestrator explicitly tells each service what to execute via commands. The orchestrator persists its state and manages the entire flow. Example: CreateOrderSaga orchestrator commands Order Service (createOrder) → Payment Service (reserveCredit) → Inventory Service (reserveInventory). On failure, the orchestrator sends compensating commands (e.g., rejectOrder). Advantages: centralized control, easier debugging and monitoring. Disadvantages: the orchestrator is a single point of failure and can become a bottleneck.

Q: Explain why 2PC is unsuitable for microservices architecture. [6 marks]

The Two-Phase Commit (2PC) protocol, despite guaranteeing atomicity across distributed nodes, is fundamentally unsuitable for microservices for six reasons:

1. Blocking Protocol: Participants lock resources until all parties respond, blocking threads and reducing scalability — incompatible with high-throughput microservices.

2. Single Point of Failure: If the coordinator crashes during the Commit phase, participants are stuck — unable to commit or rollback — causing system-wide deadlock.

3. Poor Fault Tolerance: Network failures and node crashes are common in distributed systems. 2PC cannot handle these gracefully.

4. Performance Bottleneck: The protocol adds significant latency and slows the entire system.

5. Tight Coupling: Microservices must be independent. 2PC creates hard cross-service dependencies violating this principle.

6. No Native Broker Support: Standard microservices tools — Kafka, RabbitMQ, and REST — do not support distributed 2PC natively.

The Saga Pattern is the recommended alternative, using local transactions with compensating transactions to achieve eventual consistency without distributed locks.

Q: What is DevOps and why was it introduced? Describe the DevOps Lifecycle. [10 marks]

Definition: DevOps is a software development strategy that bridges the gap between Developers and the Operations team into one integrated unit. It combines people, process, and automation to enable continuous delivery of high-quality software.

Why DevOps? Waterfall methodology was rigid with no going back between phases. Agile addressed the customer-developer gap but Operations teams still slowed release frequency. DevOps resolves this last-mile gap by making Dev and Ops one team — improving communication, automating workflows, and enabling small, frequent deployments.

DevOps Lifecycle (8 stages):

1. Code — developers write code in small chunks (tools: Git). 2. Build — compile/package into executable (Maven). 3. Test — automated testing (Selenium). 4. Release — manage and schedule builds. 5. Deploy — deploy artifacts to server (Jenkins, Ansible). 6. Operate — clients use system in real-world. 7. Monitor — track performance and uptime (Nagios, Splunk, ELK, Sensu). 8. Plan — gather feedback from monitoring for next iteration.

Key Benefits: Breaking silos, continuous improvement, minimized failures, creativity & innovation, and a performance-oriented culture.

Q: What is CQRS? When should it be used and what are its core concepts? [8 marks]

CQRS (Command Query Responsibility Segregation) is a design pattern that separates command operations (that change data) from query operations (that read data) into distinct models — often with separate databases.

Core Concepts: (1) Command Model — handles state-changing instructions (CreateOrder, UpdateCustomer). Command handlers validate inputs, enforce business rules, and emit domain events. (2) Query Model — handles read-only data retrieval, optimized for performance using Elasticsearch or denormalized SQL. It NEVER modifies state. (3) Eventual Consistency — since models are decoupled, changes propagate asynchronously via domain events or CDC. The system "eventually" becomes consistent.

When to Use: High-performance systems needing independent read/write scaling; applications with complex business logic; systems needing audit trails; domains where read and write security policies differ; microservices architectures to avoid complex JOIN queries across service boundaries.

Benefits: Scalability (scale reads and writes independently), Separation of Concerns, Performance, Security, Auditability (with event sourcing). Challenges: Increased complexity, eventual consistency lag, not needed for simple CRUD apps.
