Interaction Styles in Microservices

Before picking a communication pattern, you must understand the two dimensions of interactions:

CardinalityOne-to-OneOne request → exactly one service handles it
CardinalityOne-to-ManyOne request → multiple services handle it
TimingSynchronousClient waits (blocks) for a response
TimingAsynchronousClient does NOT block; may get response later or never

These combine into five interaction styles:

Request/Response Pattern (Synchronous)

Definition: The client sends a request and blocks until it receives a response. Both the client and service must be available simultaneously — this creates tight runtime coupling.

Implementations

REST is a set of architectural constraints built entirely on top of HTTP. A resource is the key abstraction — anything you expose to the outside world.

✅ Advantages

  • Simple and familiar
  • Testable via browser/curl/Postman
  • Firewall friendly (uses HTTP)
  • No broker needed (simpler architecture)
  • Direct request/response support

❌ Disadvantages

  • Only supports request/response style
  • Both parties must be running simultaneously (reduced availability)
  • Clients must know service URLs (needs service discovery)

SOAP uses HTTP and can use other transports (JMS, SMTP). It is an RPC mechanism (uses only POST). Supports WS-Security for message encryption and partial forwarding to different services.

Key drawback: Only POST messages, lacks HATEOAS, interfaces must be fully defined by server — very rigid.

HATEOAS is a REST constraint where the server dynamically includes links in responses, telling the client what actions are possible next.

Why it matters: Loose coupling — clients don't hardcode URLs. The server provides URLs dynamically, so changes on the server don't break clients.

Without HATEOAS: Client hardcodes /orders/123. Any URL change breaks the client.
With HATEOAS: Response includes "links": [{"rel":"cancel","href":"/orders/123/cancel"}] — client follows the link without knowing the URL in advance.

Benefits & Drawbacks of Request/Response

✅ Benefits

  • Simple and familiar
  • Easy request/reply
  • No broker needed

❌ Drawbacks

  • Reduced availability (both must be up)
  • Only request/reply style
  • Client needs service location (discovery needed)

Messaging Pattern (Asynchronous)

Core idea: Messages are exchanged via message channels. A sender writes to a channel; a receiver reads from it. They do NOT need to be available at the same time.

Message Structure

PartDescription
HeaderCollection of name-value pairs — metadata (e.g., message ID, correlation ID, reply channel)
BodyThe actual data — text or binary format

Types of Messages

TypeDescriptionExample
DocumentGeneric data; receiver decides how to interpret itReply to a command
CommandEquivalent of RPC — specifies operation + parametersPlaceOrder command
EventSignals something notable happened; usually a domain eventOrderCreatedEvent

Message Formats

FormatExamplesProsCons
TextJSON, XMLHuman-readable, self-describingLarger size (attribute names included), slower parsing
BinaryProtocol Buffers, AvroCompact, bandwidth-efficient, fasterNot human-readable

Channel Types

Point-to-Point

Delivers message to exactly ONE consumer. Used for one-to-one interactions.

Publish-Subscribe

Delivers message to ALL attached consumers. Used for one-to-many. Events typically use this.

Implementing Interaction Styles via Messaging

Client sends a command message with a reply channel header. Service processes and sends a reply with a correlation ID matching the original message ID. Client matches reply to request using the correlation ID.

Client sends a command message to service's point-to-point channel. No reply is sent or expected. Simple and straightforward.

Client publishes to a pub-sub channel. Multiple services subscribe and each receives the message. E.g., Order Service publishes to Order channel; Delivery Service also reads it.

Combines pub/sub with request/response. Client publishes with a reply channel header. Consumers write replies with correlation IDs. Client gathers responses by matching correlation IDs.

Brokerless vs. Broker-based Messaging

AspectBrokerless (e.g., ZeroMQ)Broker-based (e.g., Kafka, RabbitMQ)
ArchitectureServices talk directlyMessages go through a central broker
LatencyLower (direct)Higher (extra hop)

Bottleneck RiskNone from brokerBroker can be bottleneck

Broker Selection Factors

  • Supported programming languages
  • Standards support (AMQP, STOMP)
  • Message ordering guarantees
  • Delivery guarantees
  • Persistence (survives crashes)
  • Durability (replayed after reconnect)
  • Scalability
  • Latency
  • Competing consumers support

✅ Messaging Benefits

  • Loose runtime coupling
  • Improved availability (buffering)
  • Flexible — supports all interaction styles

❌ Messaging Drawbacks

  • Broker can be performance bottleneck
  • Broker is potential single point of failure
  • Additional operational complexity

Data Management Patterns

Each microservice has its own private database, accessible only via its API. Three implementation options for relational DBs:

  • Private tables per service — shared server, service owns specific tables
  • Schema per service — shared server, separate schema (most common)
  • Database server per service — fully separate DB server (highest isolation)

✅ Benefits

  • Loosely coupled services
  • Each service picks the best DB type (e.g., Neo4j for graphs, ElasticSearch for text)

❌ Drawbacks

  • Cross-service transactions are complex
  • Joins across databases are challenging
  • Managing multiple DB types is complex

All services share a single database with free access to each other's tables using local ACID transactions.

✅ Benefits

  • Familiar ACID transactions
  • Simpler to operate

❌ Drawbacks

  • Development-time coupling (schema changes affect all)
  • Runtime coupling (locks can block other services)
  • One DB may not fit all services

A distributed system can guarantee at most 2 of 3:

  • Consistency — all nodes see same data at same time
  • Availability — every request gets a response (success or failure)
  • Partition Tolerance — system continues despite network failures
RDBMS: Gives up P → achieves C + A. NoSQL: Must have P → choose C or A.

Distributed Transaction Protocols

Goal: Guarantee atomicity across distributed nodes — all commit or all abort.

Roles: Coordinator (initiates, decides) + Participants (vote, execute)

1
Phase 1 — Prepare (Voting): Coordinator sends PREPARE to all. Each participant executes up to commit point, writes to recovery log, replies VOTE-COMMIT or VOTE-ABORT.
2
Phase 2 — Commit/Abort: If ALL voted COMMIT → Coordinator sends COMMIT. Any VOTE-ABORT → Coordinator sends ABORT. Participants execute and release locks, send ACK.
Why 2PC fails for microservices: Blocking protocol (locks resources until all respond), coordinator = single point of failure, not supported by Kafka/RabbitMQ/REST, tight coupling, poor fault tolerance.

Problem it solves: In 2PC, if coordinator + a participant both fail during Commit phase, remaining participants block indefinitely. 3PC adds a PreCommit phase to allow safe recovery.

1
CanCommit (Vote): Coordinator asks "Can you commit?" Participants reply YES/NO. Any NO → immediate ABORT.
2
PreCommit (Prepare): If all said YES → Coordinator sends PreCommit. Participants write to log, enter 'prepared' state, send ACK. KEY: All participants now KNOW a commit is coming.
3
DoCommit (Commit): Coordinator sends DoCommit. Participants make changes permanent, release locks, send ACK. If coordinator fails here, participants can safely commit independently.

Saga Pattern

Context: Database per Service means cross-service transactions need a different mechanism. 2PC/3PC are not viable.

Definition: A saga is a sequence of local transactions. Each service executes its local transaction and publishes an event/message to trigger the next step. If any step fails, compensating transactions are executed to undo previous steps.

Two Coordination Approaches

Choreography (Event-Driven)

  • No central controller
  • Services react to events autonomously
  • Each publishes events, others subscribe
  • Loose coupling
  • No single point of failure
  • Hard to debug, flow is decentralized
  • Complex failure handling per service

Orchestration (Centralized)

  • Central orchestrator manages the saga
  • Tells each service what to do
  • Clear business logic in one place
  • Easy to debug/monitor
  • Orchestrator = single point of failure
  • 'God service' anti-pattern risk
  • Services dependent on orchestrator

Choreography Example — Order Placement

1
Order Service creates order (PENDING) → publishes OrderCreatedEvent
2
Payment Service listens → charges customer → publishes PaymentCompletedEvent
3
Inventory Service listens → reserves stock → publishes InventoryReservedEvent
Failure path: If Payment fails → emits PaymentFailedEvent → Order Service listens and cancels order (compensation). If Inventory fails → emits InventoryFailedEvent → Payment Service issues refund.

Orchestration Example — CreateOrderSaga

1
Orchestrator sends createOrder() → Order Service sets PENDING, replies
2
Orchestrator sends reserveCredit() → Payment Service checks credit
3
Orchestrator sends reserveInventory() → Inventory Service reserves, saga complete (APPROVED)
Failure: If creditLimitExceeded → Orchestrator sends rejectOrder() to Order Service → REJECTED state.

Saga API Design Challenges

OptionHowTrade-off
SynchronousRespond when saga completesSpecifies outcome, but reduced availability (client waits)
Asynchronous (Recommended)Respond immediately after saga creationBetter availability, but client must poll/be notified

Saga Orchestrator: A persistent object that tracks saga state and invokes participants. Loads state from DB on each reply, determines next step, invokes participant, persists updated state.

CQRS — Command Query Responsibility Segregation

Core Idea: Separate Commands (write/change data) from Queries (read data) into different models and even different databases.

Problem it solves: In monolithic apps, one DB handles both complex joins AND CRUD. As complexity grows, this becomes unmanageable — long queries lock the DB, complex validations slow writes.

Core Concepts

ConceptDescription
Command ModelHandles instructions that change state (CreateOrder, CancelInvoice). Command handlers validate, enforce business rules, emit domain events.
Query ModelHandles data retrieval only. Optimized for reads (denormalized tables, Elasticsearch). Never modifies state. Returns DTOs (JSON).
Eventual ConsistencyWrite and read models are decoupled. Changes propagate asynchronously via domain events or CDC. Read model may lag slightly.

CQRS with Kafka (Manual Event Publishing)

  • Service explicitly emits events to Kafka after a command
  • Full control over event payload and content
  • Risk of dual-write inconsistency (DB + Kafka)
  • Requires outbox pattern for reliability

CQRS with Kafka + CDC (Debezium)

  • Service only writes to DB; Debezium captures changes and emits events automatically
  • No manual event publishing needed
  • Events are low-level (row changes, not rich business events)
  • Cannot emit events before committing DB transaction

✅ CQRS Benefits

  • Independent read/write scaling
  • Better performance (optimized models)
  • Separation of concerns
  • Different security per operation
  • Auditability (with event sourcing)

❌ CQRS Challenges

  • Increased complexity
  • Data duplication and sync overhead
  • Eventual consistency lag
  • Overkill for simple CRUD apps

Master Summary
  • 🔗 Communication Patterns Overview
    • Two dimensions: Cardinality (one-to-one / one-to-many) × Timing (sync / async)
    • 5 styles: Request/Response, Async Req/Res, One-way Notification, Pub/Sub, Pub/Async Responses
    • Synchronous = tight coupling; Async = loose coupling
  • 🔄 Request/Response (Sync)
    • REST: built on HTTP, resource-centric, firewall-friendly, no broker, needs service discovery
    • SOAP: HTTP + other transports, RPC-style (POST only), WS-Security support, lacks HATEOAS
    • HATEOAS: server provides links dynamically → loose coupling → clients don't hardcode URLs
    • Benefits: simple, no broker | Drawbacks: both must be up, only req/res style, needs discovery
  • 📨 Messaging (Async)
    • Message = Header (metadata) + Body (data)
    • Types: Document, Command, Event
    • Formats: Text (JSON/XML — readable, larger) | Binary (Protobuf/Avro — compact, faster)
    • Channels: Point-to-Point (one consumer) | Pub-Sub (all consumers)
    • Brokerless (ZeroMQ): lower latency, both must be up | Broker-based (Kafka/RabbitMQ): buffers, decoupled
    • Benefits: loose coupling, availability, flexible | Drawbacks: broker bottleneck/SPOF, operational complexity
  • 🗄️ Data Management Patterns
    • DB per Service: private DB per service, chosen per need, but cross-service joins/transactions hard
    • Shared DB: ACID-easy, simple, but dev/runtime coupling, lock conflicts
    • CAP: only 2 of 3 (C/A/P). SQL = C+A (no P). NoSQL = C+P or A+P
    • ACID (SQL): strict | BASE (NoSQL): optimistic, eventual consistency
    • NoSQL types: Document (MongoDB), Column (Cassandra), Key-Value (Redis), Graph (Neo4j)
  • ⚙️ Distributed Transactions
    • 2PC: Prepare → Commit/Abort. Blocking, SPOF coordinator, not for microservices
    • 3PC: CanCommit → PreCommit → DoCommit. Non-blocking recovery, but more RTTs, not partition-safe
    • Saga: sequence of local transactions + compensating transactions for rollback
    • Choreography: event-driven, decentralized, no SPOF | Orchestration: central control, easy to debug, SPOF risk
    • Saga API: Async response recommended (better availability)
  • 📊 CQRS
    • Separate Command (write) and Query (read) models
    • Can use different DBs: NoSQL for reads, RDBMS for writes
    • Eventual consistency between write and read sides
    • With Kafka (manual): full control | With Kafka+CDC (Debezium): automatic, low-level events
    • Benefits: scaling, performance, security | Drawbacks: complexity, sync overhead, latency

"Commands Change, Queries Question" — Commands write/change state, Queries only read

⚠️
Common Exam Traps & Mistakes
Trap 1: Confusing "synchronous" with "one-to-one." Synchronous means the client WAITS — it's about timing, not cardinality.
Trap 2: Thinking REST supports multiple interaction styles. REST only supports Request/Response.
Trap 3: Confusing "point-to-point" and "pub-sub." Point-to-point → ONE consumer. Pub-sub → ALL consumers.
Trap 4: 3PC is NOT the solution for microservices — it's still impractical (not partition-safe, rarely used in production).
Trap 5: CQRS does NOT mean two separate databases are mandatory — but they are recommended for performance.
Trap 6: Saga rollback is NOT like SQL ROLLBACK. It requires writing compensating transactions at the application level.
Trap 7: Choreography has NO single point of failure. Orchestration DOES (the orchestrator itself).
Trap 8: CDC (Debezium) captures row-level changes — NOT rich domain business events.
Trap 9: Graph databases store relationships explicitly — they do NOT compute them at query time (unlike SQL joins).
Trap 10: HATEOAS provides loose coupling by letting the SERVER control URL structure — not the client.

🌙 Night Before Exam — Key Rules

  • Sync = blocks client. Async = non-blocking.
  • REST = HTTP only, resource-based, no broker needed, only req/response style.
  • HATEOAS = server provides URLs dynamically → loose coupling.
  • Message = Header (metadata) + Body (data). Types: Document / Command / Event.
  • Point-to-Point = one consumer. Pub-Sub = all consumers.
  • CAP: pick 2 of 3. SQL = C+A. NoSQL must pick C or A (always has P).
  • ACID = SQL (pessimistic). BASE = NoSQL (optimistic, eventually consistent).
  • 2PC: PREPARE → COMMIT/ABORT. Blocking, SPOF, NOT for microservices.
  • 3PC: CanCommit → PreCommit → DoCommit. Non-blocking but not partition-safe.
  • Saga = sequence of local transactions + compensating transactions for rollback.
  • Choreography = events, decentralized, no SPOF. Orchestration = central, easier debug, SPOF.
  • CQRS = Commands write, Queries read. Separate models. Eventual consistency between them.
  • CDC (Debezium) = captures DB row changes automatically for event streaming.
  • Graph DB: relationships stored explicitly (not computed). Neo4j. Very fast traversal.

📦 NoSQL Quick Reference

  • MongoDB — Document DB
  • Cassandra — Column DB (heavy writes)
  • Redis — Key-Value (sessions, caching)
  • Neo4j — Graph DB (social, fraud)

  • Async Req/Res — 1:1, Async
  • One-way Notification — 1:1, Async
  • Pub/Sub — 1:Many, Async
  • Pub/Async Responses — 1:Many, Async
