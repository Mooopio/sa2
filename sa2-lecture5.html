شرح كل Stage بعمق

كل مرحلة بتدور في الـ Loop المستمر — دي أهم حاجة تفهمها 👇

🧭
1️⃣ Plan — التخطيط
🟡 البداية وهي في نفس الوقت النهاية
🔹 الجملة: gathers information from monitoring stage
👉 التخطيط بيعتمد على: Feedback من المستخدم + Data من Monitoring
🔥 الفكرة العميقة:
مش بنخطط من دماغنا
👉 بنخطط بناءً على واقع فعلي

الفكرة: نحدد هنعمل إيه + ناخد Feedback من Monitoring
✔ تحسين مستمر
💻
2️⃣ Code — البرمجة
🟡 المطورين يكتبوا الكود
🔹 الجملة: developers build the code
👉 المطورين يكتبوا الكود
🔧 Tools: Git

✔ version control   ✔ teamwork

💡 الفكرة:
👉 الكود بيتكتب على شكل: Small changes — مش Project كبير مرة واحدة
🏗️
3️⃣ Build — البناء
🟡 نحول الكود لبرنامج قابل للتشغيل
🔹 الجملة: build the version of their program
👉 نحول الكود لـ: Executable + Package
🔧 Tools: Maven Gradle

✔ compile + package

💡 مثال: Java code → JAR file
🧪
4️⃣ Test — الاختبار
🟡 اكتشاف bugs بدري
🔹 الجملة: testing must be automated
👉 الاختبار لازم يكون: ❌ Manual · ✅ Automated
🔧 Tools: Selenium

✔ اكتشاف bugs بدري

🔥 الفكرة:
👉 كل تغيير يتعمله Test أوتوماتيك
علشان: نكتشف Bugs بسرعة
🚦
5️⃣ Release — الإصدار
🟡 تجهيز النسخة للنشر + تنظيم الإصدارات
🔹 الجملة: managing and scheduling build before deployment
👉 مرحلة تنظيم: إمتى هننزل؟ + فين هننزل؟
💡 دي مرحلة: Control + Approval
✔ إدارة Versions
🚀
6️⃣ Deploy — النشر
🟡 رفع البرنامج على السيرفر
🔹 الجملة: deploy code on the server
👉 ننزل التطبيق على السيرفر
🔥 الفرق:
Release = تجهيز
Deploy = تنفيذ فعلي

✔ user يقدر يستخدمه
⚙️
7️⃣ Operate — التشغيل
🟡 تشغيل التطبيق في الواقع
🔹 الجملة: application is run
👉 السيستم شغال فعليًا · 👤 المستخدم بيستخدمه
✔ المستخدمين يستخدموه
📊
8️⃣ Monitor — المراقبة
🟡 متابعة الأداء + اكتشاف المشاكل
🔹 الجملة: ensure uptime and performance
👉 نراقب: هل السيستم شغال؟ + الأداء عامل إيه؟
🔧 Tools: Nagios Splunk ELK

✔ uptime   ✔ performance

🔥 الفكرة:
👉 أي مشكلة → ترجع لـ Plan تاني
💣 وده سر DevOps

💥 الصورة الكاملة — ربط كل حاجة ببعض

Monitor → نكتشف مشكلة

👉 بيحسن نفسه بنفسه

1️⃣ Automation — كل حاجة أوتوماتيك
2️⃣ Feedback Loop — كل مرحلة تأثر على اللي بعدها

Continuous Deployment (CD)

نشر الكود تلقائيًا بعد ما ينجح في الاختبارات

💡 التعريف البسيط:
Continuous Deployment = نشر الكود تلقائيًا بعد ما ينجح في الاختبارات
Code → Build → Test → (لو كله OK) → Deploy تلقائي 🚀
💥 بدون أي تدخل بشري

🧠 يعني إيه عمليًا؟

  • المطور يكتب كود
  • يرفعه على Git
  • النظام (CI/CD pipeline): يعمل Build
  • يعمل Test
  • لو كله تمام 👇  → ينزل مباشرة على Production
🔹 الجملة: automates deploying changes to staging or production
👉 يعني: 📌 أي كود جديد يتعمله: Merge ➡️ يتنزل أوتوماتيك على السيرفر
🔥 الجملة: save time and effort
👉 بدل: Deployment يدوي أو تستنى schedule ➡️ كله يحصل فورًا
