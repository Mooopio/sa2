Messaging Patterns

A complete deep-dive into asynchronous communication, message formats, channel types, interaction styles, and broker architectures. Optimized for exam success.

1. What is Asynchronous Messaging?

Core Idea: Instead of one service calling another directly and waiting for a reply (synchronous), asynchronous messaging uses channels as intermediaries. A sender drops a message into a channel and moves on — it does not block and wait.

Real-world analogy: Think of it like email. When you send an email, you don't stand still waiting for a reply. You continue with your day. The receiver reads it when they're available and responds when ready. That's asynchronous messaging.

Message Structure

Every message has two parts:

PartWhat It ContainsAnalogy
HeaderMetadata — name-value pairs (e.g. correlation ID, reply channel, timestamp)The envelope of a letter

2. Types of Messages

1. Document Message

A generic message containing only data. The receiver decides how to interpret it. A reply to a command is typically a document message.

Analogy: A report handed to a manager — it contains information, and the manager decides what to do with it.

2. Command Message

Equivalent to an RPC (Remote Procedure Call) request. It specifies the operation to invoke and its parameters. It tells the receiver what to do.

Analogy: A work order — "Please process this refund for customer #1234."

3. Event Message

Indicates that something notable has occurred in the sender. Often a domain event representing a state change (e.g., Order placed, Customer registered).

Analogy: A news broadcast — "Something happened, in case you care." Listeners who are interested react; others ignore it.

3. Message Formats

Choosing the right format affects performance, readability, and compatibility.

Advantages:

  • Human-readable — easy to debug and inspect
  • Self-describing — field names are included in the message
  • Widely supported across all programming languages

Disadvantages:

  • Overhead from attribute names: Every field name is repeated in every message (e.g., "orderId": 123 — the key "orderId" takes space)
  • Parsing overhead: Converting text to objects is CPU-intensive, especially for large messages
⚠ Exam Trap The "self-describing" advantage is the flip side of the "overhead" disadvantage. The same property (field names in the message) is both an advantage AND a disadvantage.

Binary formats organize the bit stream around data structures, not characters. Field names are NOT included in the message — they're encoded as numbers.

  • More compact — saves bandwidth
  • Faster to parse — no text conversion needed
  • NOT human-readable — harder to debug
  • Require a schema to interpret
💡 When to choose binary? When efficiency and performance are critical — high-frequency trading, IoT data streams, inter-datacenter communication.

4. Types of Channels

Channel TypeDeliveryInteraction StyleExample Use
Point-to-Point Exactly one consumer One-to-one Send a command to a specific service
Publish-Subscribe All attached consumers One-to-many Broadcasting an event (OrderPlaced)

Point-to-Point analogy: A direct phone call to one person.

Pub/Sub analogy: A radio broadcast — everyone tuned to that station hears the same message.

5. Interaction Styles Using Messaging

① Asynchronous Request/Response

The client and service exchange a pair of messages:

  1. Client sends a command message to a point-to-point channel owned by the service. The command includes a reply channel header (where to send the reply).
  2. Service processes the request and sends a reply message to the client's reply channel, including a correlation ID matching the original message ID.
  3. Client uses the correlation ID to match the reply with the original request.
🔑 Key Fields to Remember Reply Channel Header — tells the service where to send the reply.
Correlation ID — links the reply to the original request (same value as the message identifier).

② One-Way Notification

The simplest pattern. Client sends a command message to a point-to-point channel. The service processes it and sends NO reply.

Example: Sending a "send welcome email" notification — you don't need a reply confirming the email was sent.

③ Publish/Subscribe

A client publishes a message to a publish-subscribe channel. Multiple consumers are subscribed and each receives the message independently.

Example: The Order Service publishes OrderPlaced events to an Order channel. The Delivery Service, Inventory Service, and Notification Service all subscribe and react independently.

④ Publish/Async Responses (Hybrid)

A higher-level pattern combining pub/sub + request/response:

  1. Client publishes to a pub/sub channel with a reply channel header
  2. Multiple consumers process and each writes a reply message with a correlation ID to the reply channel
  3. Client collects all replies using the correlation ID

Use case: Asking multiple services for quotes, waiting for all responses (e.g., flight booking aggregators).

6. Message Broker Architectures

Brokerless Messaging

Services communicate directly with each other — no intermediary.

Technology: ZeroMQ — supports TCP, UNIX-style domain sockets, multicast.

AdvantagesDisadvantages
Lower latency (no broker hop)Services must know each other's network locations (discovery needed)
No broker = no single point of failureReduced availability (both sender AND receiver must be up)
Less operational complexityGuaranteed delivery is harder to implement
Lighter network traffic

Broker-Based Messaging

All messages flow through an intermediary message broker. The sender writes to the broker; the broker delivers to the receiver.

Key benefits:

  • Location transparency: Sender doesn't need to know the consumer's network location
  • Message buffering: Broker stores messages until the consumer is ready to process them (improves availability)

Popular brokers: ActiveMQ, RabbitMQ, Apache Kafka, AWS Kinesis, AWS SQS

⚠ Trade-off Reality Low-latency brokers often sacrifice ordering guarantees and persistence. High-reliability brokers (guaranteed delivery, disk persistence) usually have higher latency. There is no "best" broker — it depends on your application's requirements.

🔷 Asynchronous Messaging Fundamentals

  • Messages exchanged over channels; sender writes, receiver reads — no blocking
  • Message = Header (metadata/name-value pairs) + Body (data: text or binary)

3 Message Types:

  • Document — generic data; receiver interprets; used for replies
  • Command — like RPC; specifies operation + parameters
  • Event — something notable happened; often a domain event (state change)

🔷 Message Formats

  • Text (JSON/XML): Human-readable, self-describing | Disadvantages: attribute name overhead, parsing cost
  • Binary (Protocol Buffers, Avro): Compact, fast | Disadvantages: not human-readable, requires schema
  • Use binary when efficiency and performance are critical

🔷 Channel Types

  • Point-to-Point → exactly one consumer → one-to-one interaction
  • Publish-Subscribe → all attached consumers → one-to-many interaction

🔷 4 Interaction Styles

  1. Async Request/Response — command message + reply channel header → reply with correlation ID → client matches by correlation ID
  2. One-Way Notification — command message → point-to-point channel → no reply sent
  3. Publish/Subscribe — event to pub/sub channel → all subscribers react
  4. Publish/Async Responses — pub/sub + reply channel header → multiple replies collected via correlation ID

🔷 Broker Architectures

Brokerless (e.g., ZeroMQ):

  • ✅ Lower latency, no bottleneck, less complexity
  • ❌ Must know locations, both must be up, hard to guarantee delivery

Broker-Based (e.g., Kafka, RabbitMQ):

  • ✅ Location transparency, message buffering, flexible
  • ❌ Possible bottleneck, possible SPOF, operational complexity

🔷 Broker Selection Factors (9)

  • Programming language support
  • Messaging standards (AMQP, STOMP)
  • Message ordering
  • Delivery guarantees
  • Persistence (disk survival)
  • Durability (missed messages after reconnect)
  • Scalability
  • Latency
  • Competing consumers support

Benefits: Loose coupling · Improved availability · Flexible communication

Drawbacks: Performance bottleneck risk · Single point of failure risk · Operational complexity

🔷 Data Management Patterns (Brief)

  • Database per service — each service has its own DB
  • Shared database — services share a DB
  • SAGA — sequence of local transactions for cross-service consistency
  • CQRS — materialized views for efficient queries
  • Event sourcing — persist aggregates as sequence of events
  • Domain event — publish event whenever data changes

⚠️ Common Exam Traps

Trap 1: Correlation ID vs Reply Channel

These are two different things. The reply channel header tells the service WHERE to send the reply. The correlation ID is how the client identifies WHICH request a reply belongs to. Don't mix them up.

Trap 2: Brokerless = Better?

NOT necessarily. Brokerless has lower latency but lower availability and no guaranteed delivery. It's a trade-off. The exam often asks you to compare — always mention both sides.

Trap 3: Self-Describing = Always Good

Self-describing (text formats) is also the source of overhead. The same feature that makes JSON readable is what makes it larger and slower. It's simultaneously an advantage and a disadvantage.

Trap 4: "One Broker Fits All"

The lecture explicitly states: "different parts of your application will have different messaging requirements." Never say one broker is universally best.

Trap 5: Pub/Sub vs Async Publish/Response

Pub/Async Responses = Pub/Sub + Request/Response combined. The difference is that in Pub/Async, the publisher includes a reply channel header and collects responses via correlation ID. Pure pub/sub doesn't expect replies.
