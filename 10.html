🗂️ Overview: The 6 Data Management Patterns

In a microservices architecture, each service is an independent unit. The biggest challenge is: How do we manage data when services are isolated? These 6 patterns solve that problem:

Database per Service Shared Database SAGA CQRS Event Sourcing Domain Event
💡Think of a microservices system like a city of shops. Each shop (service) has its own storage room. The patterns define how the shops share information, keep records in sync, and handle transactions across multiple shops.
1️⃣ Database per Service Pattern▼
What is it?

Each microservice has its own private database, accessible only through that service's API. No other service can directly query or write to it.

🏪Analogy: Imagine Amazon, where the Orders department has its own filing cabinets, and the Customers department has its own separate cabinets. One department cannot open the other's drawers — they must ask each other formally via phone (API).

Three implementation options (know all three!):

  • Private-tables-per-service: All services share one DB server, but each service owns specific tables. Lowest overhead but weakest isolation.
  • Schema-per-service: Each service gets its own database schema. Clearer ownership. Good balance of isolation and simplicity.
  • Database-server-per-service: Each service has a completely separate DB server. Highest isolation. Used for high-throughput services.
✅ Benefits
  • Loose coupling: Changes in one service's DB don't break other services.
  • Technology freedom: Each service can use the best DB type for its needs. (e.g., ElasticSearch for text search, Neo4j for social graphs)
❌ Drawbacks
  • Distributed transactions: Business operations spanning multiple services are hard (CAP theorem limits this).
  • Cross-service joins: Querying data across multiple databases is complex.
  • Operational complexity: Managing many different databases (SQL + NoSQL) increases DevOps overhead.
⚖️ CAP Theorem (Critical Exam Topic)▼
The CAP Theorem

It is impossible for a distributed system to simultaneously guarantee all three properties:

PropertyMeaning
Consistency (C)All nodes see the same data at the same time
Availability (A)Every request gets a response (success or failure)
Partition Tolerance (P)System keeps running even if network splits
⚠️A distributed system can satisfy only 2 of the 3 at a time. You MUST choose which one to sacrifice.
DB TypeChoiceTrade-off
RDBMS (SQL)C + A (no P)Not distributed; no network partitions assumed
NoSQL (want P)C + P or A + PDrop A → wait for consistency; Drop C → accept stale data
🧠 Mnemonic
CAP = Can't All Please
Can't (Consistency) All (Availability) Please (Partition tolerance) — You can only satisfy two at once!

2️⃣ Shared Database Pattern▼
What is it?

Multiple services share a single common database. Any service can freely access data owned by another service directly using local ACID transactions.

⚠️Anti-pattern alert: This is generally considered an anti-pattern in microservices because it tightly couples services. But it has legitimate use cases for small systems.

Benefits:

  • Developers use familiar ACID transactions for data consistency.
  • Simpler to operate — just one database to manage.

Drawbacks:

  • Development-time coupling: Changing a schema requires coordination with all teams using that schema.
  • Runtime coupling: A long-running transaction in Service A can lock tables and block Service B.
  • Technology lock-in: One DB may not fit the needs of all services.
🗄️ NoSQL Database Types (4 Types — High Exam Weight)▼
📄 1. Document Database

Structure: Collections → Documents → Key-Value Pairs. Each document has a unique key and flexible content.

Best for: Content management systems, blogging platforms, e-commerce, real-time analytics, flexible schema queries.

Avoid when: Queries against varying aggregates.

Example: MongoDB

📊 2. Column Database

Structure: Keyspace → Column Families → Rows → Columns (key-value pairs). Each row is identified by a unique row key. Different rows can have different numbers of columns.

Best for: Heavy write operations, large data volumes, OLAP queries, high performance/availability/scalability, real-time analytics, web analytics.

Avoid when: Query patterns change frequently; early-stage development.

Example: Cassandra

🔑 3. Key-Value Database

Structure: Simplest NoSQL model. Just pairs of key → value. Keys are unique identifiers; values can be simple or complex. Retrieve values using atomic keys.

Best for: Session data, user profiles, preferences, blog comments, product recommendations. High read/write velocity without frequent updates.

Avoid when: (1) Multiple unique key operations needed. (2) Relationships between values needed (joins). (3) Queries by specific data fields. (4) Part of the value needs frequent updates.

Example: Redis

🕸️ 4. Graph Database

Structure: Composed of:

  • Nodes: Main entities (like rows in RDBMS)
  • Edges (Relationships): Connections between nodes (like foreign keys in RDBMS)
  • Labels: Attributes that group similar nodes
  • Properties: Key/value pairs stored within nodes or relationships

Key Differentiator: Relationships are first-class citizens — stored explicitly, not calculated at query time (unlike RDBMS joins). Traversal is very fast.

Best for: Social networks, recommendation systems, fraud detection, bioinformatics, security/access control, network/cloud management, complex graph-based queries.

Avoid when: Connections between data are NOT the primary focus, and data is transactional in nature.

Example: Neo4j

🔍In RDBMS, relationships are stored in foreign key fields or junction tables, and joins are calculated at query time — expensive! In graph DBs, relationships are persisted directly and traversal is instant.

🔗 Other Patterns Overview (SAGA, CQRS, Event Sourcing, Domain Event)▼
SAGA Pattern

A sequence of local transactions used to maintain data consistency across services. Instead of a single distributed transaction, each service performs a local transaction and publishes an event/message to trigger the next step. If a step fails, compensating transactions are executed to undo previous steps.

CQRS (Command Query Responsibility Segregation)

Implement queries by maintaining one or more materialized views that can be efficiently queried. Separates the "read" model (Query) from the "write" model (Command). Solves the problem of joining data across multiple microservice databases.

Event Sourcing

Instead of storing the current state of data, store a sequence of events (the history of changes). The current state is reconstructed by replaying these events.

Domain Event

Publish an event whenever data changes. Other services can subscribe and react to these events, enabling loose coupling and real-time data synchronization.

🗂️ 6 Data Management Patterns
  • Database per Service: Each service → its own private DB, only via API
  • Shared Database: All services share one DB (anti-pattern in most cases)
  • SAGA: Sequence of local transactions for cross-service consistency
  • CQRS: Materialized views for efficient queries; separates read/write
  • Event Sourcing: Store events as history, not current state
  • Domain Event: Publish events when data changes
🗄️ DB per Service — 3 Implementation Options
  • Private tables: Same server, different tables → lowest overhead
  • Schema per service: Same server, different schemas → clearer ownership
  • DB server per service: Separate servers → highest isolation, for high-throughput
⚖️ CAP Theorem — 3 Properties, Choose 2
  • Consistency (C): All nodes see same data at same time
  • Availability (A): Every request gets a response
  • Partition Tolerance (P): System works despite network splits
  • RDBMS = C + A (no P) | NoSQL = P + (C or A)
🧪 ACID vs BASE
  • ACID: Atomic, Consistent, Isolated, Durable → SQL, pessimistic, immediate consistency
  • BASE: Basically Available, Soft state, Eventually consistent → NoSQL, optimistic, scalable
🗄️ 4 NoSQL Types + Key Facts
  • Document (MongoDB): Collections → Documents → KV. Best: CMS, e-commerce, blogs.
  • Column (Cassandra): Keyspace → Column Families → Rows. Best: OLAP, heavy writes, analytics.
  • Key-Value (Redis): Key → Value. Best: sessions, caching, profiles. Avoid: joins, relations.
  • Graph (Neo4j): Nodes + Edges + Labels + Properties. Relationships stored explicitly. Best: social networks, fraud detection.
🔑 Graph DB Key Distinction
  • Relationships are first-class citizens — stored explicitly, not computed at query time
  • RDBMS uses foreign keys/junction tables + expensive join operations
  • Graph DB stores relationships as persisted data → instant traversal
